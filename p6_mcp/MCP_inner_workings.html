MCP Inner Workings — Complete Reference

How the Model Context Protocol works end-to-end: the three primitives, schema discovery, and the full tool call lifecycle between Client Agent, LLM, and MCP Server.

The Three Layers

┌─────────────────────────────────┐ │ LLM (Claude, GPT-4o) │ ← does reasoning, decides to call tools ├─────────────────────────────────┤ │ MCP Client / Host App │ ← Claude Desktop, your Python script, │ (client_intro.py) │ VS Code extension, agent code ├─────────────────────────────────┤ │ MCP Server (server_intro.py) │ ← your FastMCP server └─────────────────────────────────┘
Key point: The LLM and MCP Server never talk directly. The Client Agent sits between them, translating between the Anthropic API protocol and the MCP protocol. The LLM only sees schemas — it never touches the MCP server.

The Three MCP Primitives

An MCP server exposes three types of capabilities. All three are discovered via schema — the client calls a list method and gets back metadata describing what's available.

PrimitiveWhat it isWho triggers itDiscovery callInvocation call
ToolsCallable functions that perform actions or return dataLLM autonomously during reasoningsession.list_tools()session.call_tool(name, args)
ResourcesReadable data (files, DB rows, URIs) for contextClient explicitly, before/outside LLM conversationsession.list_resources()session.read_resource(uri)
Prompt TemplatesNamed, reusable prompt strings stored on the serverClient fetches and injects into LLM messagesession.list_prompts()session.get_prompt(name, args)

Tools vs Resources — the key distinction

Both can return data. The difference is who calls them and when:

ToolResource
Triggered byLLM (autonomous, mid-conversation)Client (explicit, before conversation)
Args passed asJSON payloadEmbedded in URI path
AnalogyFunction call / API endpointHTTP GET / file read
LLM awarenessLLM sees schema, decides to callLLM sees data already in context — never "calls" it
Use forLive data, actions, side effectsStatic/semi-static context to pre-load
In practice, you could replace resources with tools and it would still work. Resources exist as a first-class primitive because host applications like Claude Desktop show them differently — as browsable context the user can explicitly load, not callable functions the LLM invokes.

Schema — How FastMCP Builds It

When FastMCP starts, it introspects all decorated functions and builds schemas automatically from type hints and docstrings. You never write JSON schemas manually.

# Your Python function
@mcp.tool()
def get_stock_price(ticker: str) -> str:
    """Get the current stock price for a given ticker symbol."""
    ...

# FastMCP generates this schema automatically:
{
  "name": "get_stock_price",
  "description": "Get the current stock price for a given ticker symbol.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "ticker": { "type": "string" }
    },
    "required": ["ticker"]
  }
}

The Full Tool Call Lifecycle

A user asks "What's Apple's stock price?" — here's every interaction between all three layers:

Phase 1 — Client discovers tools from MCP Server

Client Agent MCP Server | | |--- list_tools() ------------------>| | | scans @mcp.tool() decorators | | builds schemas from type hints + docstrings |<-- [{name: "get_stock_price", | | description: "...", | | inputSchema: { | | ticker: {type: "string"} | | }}] |

Client holds tool schemas in memory. MCP Server's job is done for now.

Phase 2 — Client sends question + schemas to LLM

Client Agent LLM (Claude API) | | |--- messages.create() ------------>| | user: "What's AAPL price?" | | tools: [{ | | name: "get_stock_price", | ← MCP schema translated to | description: "...", | Claude's tools[] format | input_schema: { | | ticker: {type: "string"} | | } | | }] |
The client translates MCP schemas into the LLM's native format. For Claude this means the Anthropic tools=[] parameter. The LLM never talks to the MCP server — it only sees schemas the client hands it.

Phase 3 — LLM responds with a tool call instruction

Client Agent LLM (Claude API) | | |<-- response -----------------------| | stop_reason: "tool_use" | | content: [{ | | type: "tool_use", | | id: "toolu_abc123", | | name: "get_stock_price", | ← LLM picked this tool | input: { | | ticker: "AAPL" | ← LLM filled in args from context | } | | }] |
The LLM does NOT call anything. It returns a JSON instruction saying "call this tool with these args." It's just text. The client is responsible for acting on it.

Phase 4 — Client maps LLM instruction to MCP tool call

Client Agent MCP Server | | | reads: name="get_stock_price" | | input={"ticker": "AAPL"} | | | |--- call_tool( | | "get_stock_price", | | {"ticker": "AAPL"} | | ) ------------------------------>| | | looks up function by name | | calls get_stock_price(ticker="AAPL") | | runs yfinance, gets price |<-- "Apple Inc. (AAPL): $271.06" --|

Is the Phase 4 mapping hardcoded?

No — it's completely generic. The client logic works for any tool:

# Generic dispatch — no hardcoding, works for ANY tool
if response.stop_reason == "tool_use":
    tool_call = response.content[0]
    result = await session.call_tool(
        tool_call.name,    # whatever name LLM returned
        tool_call.input    # whatever args LLM returned
    )

# MCP server side is also generic:
# call_tool("get_stock_price", {"ticker":"AAPL"})
#   → looks up "get_stock_price" in registry
#   → calls get_stock_price(ticker="AAPL")
The only hardcoded part is the Python function itself on the server. Everything in between — LLM schema format, client dispatch, MCP routing — is generic infrastructure. This is why MCP is valuable as a standard: write the function once, the entire discovery-to-execution chain is automatic.

Phase 5 — Client sends tool result back to LLM

Client Agent LLM (Claude API) | | |--- messages.create() ------------>| | [original messages] | | + assistant: tool_use message | | + user: { | | type: "tool_result", | | tool_use_id: "toolu_abc123",| | content: "Apple: $271.06" | | } | | | |<-- "Apple Inc is trading at $271" -|

Client sends full conversation history including the tool result. LLM now has the data it needed and generates a natural language response.

Complete Flow Summary

User: "What's Apple's stock price?" ↓ Client fetches schemas from MCP Server [MCP protocol] ↓ Client sends question + schemas to LLM [Anthropic API] ↓ LLM returns tool_use instruction (JSON only) [Anthropic API] ↓ Client calls MCP Server with name + args [MCP protocol] ↓ MCP Server runs Python function, returns data [Python] ↓ Client sends result back to LLM [Anthropic API] ↓ LLM generates final natural language answer [Anthropic API] ↓ User sees: "Apple Inc is trading at $271.06"

Resource Lifecycle — How Client, MCP Server, and LLM Interact

Unlike tools (where the LLM drives the call), resources are driven entirely by the client application. The LLM never calls a resource — it just sees the data already in its context.

Step 1 — Client fetches resource content from MCP Server

Client Agent MCP Server | | |--- read_resource( | | "stocks://earnings/AAPL" | | ) ------------------------------>| | | extracts ticker="AAPL" from URI | | calls get_earnings_note("AAPL") |<-- "Q1 2025: Revenue $124B..." |

The client author hardcoded this URI — they already know what context the LLM will need. No discovery required for a purpose-built client.

Step 2 — Client injects resource content into LLM context

Client Agent LLM (Claude API) | | |--- messages.create() ------------>| | system: "You are a stock | | analyst. Earnings data: | | Q1 2025: Revenue $124B..." | ← resource content injected here | user: "Should I buy AAPL?" | | | |<-- "Based on Q1 earnings, AAPL..." |

The LLM reads the earnings data as part of its context. From the LLM's perspective it was just there in the prompt — it never "called" anything to get it.

So why does list_resources() exist?

If the client already knows which resources to fetch (hardcoded), why discover them at all? Because list_resources() is for generic clients that don't know the server in advance:

Client typeNeeds list_resources()?Why
Hardcoded app (your client_intro.py)No — redundantDeveloper already knows the URIs
Claude DesktopYesConnects to ANY MCP server, can't hardcode unknown URIs
Developer inspection toolYes"What does this server expose?" exploration
Dynamic UI builderYesGenerates a browsable list of resources for the user
Claude Desktop example: User installs a new MCP server. Claude Desktop calls list_resources(), gets back the URI list, shows it as a browsable file tree. User clicks a file → client calls read_resource(uri) → content injected into context. The human is the decision-maker, not code and not the LLM.

Tool vs Resource — who is in the driver's seat

TOOL flow: User question → LLM decides which tool to call (autonomous) → Client calls MCP Server → Result returned to LLM → LLM answers RESOURCE flow: Client decides what context to load (hardcoded policy OR human UI choice) → Client calls MCP Server → Content injected into context window → User question → LLM answers with context already available
The rule of thumb: If you need data mid-conversation based on what the user says → Tool (LLM decides). If you know upfront what context the LLM will need → Resource (client pre-loads it). In most simple agents you can use tools for everything — resources add value when a human or app logic, not the LLM, controls what context gets loaded.

Are Resources Redundant? — The Honest Answer

Yes — anything a resource does, a tool can do:

# As a resource
@mcp.resource("stocks://earnings/{ticker}")
def get_earnings_note(ticker: str) -> str:
    return "Q1 2025: Revenue $124B..."

# As a tool — does exactly the same thing
@mcp.tool()
def get_earnings_note(ticker: str) -> str:
    """Get earnings notes for a ticker."""
    return "Q1 2025: Revenue $124B..."

Same data, same result. And tools can also be called directly by the client — the LLM doesn't have to initiate them:

# Client hardcoding a tool call directly — perfectly valid
result = await session.call_tool("get_stock_price", {"ticker": "AAPL"})

So the distinction breaks down completely when examined:

Convention in specReality
ToolsLLM calls themClient can call them directly too
ResourcesClient calls themNothing stops client calling a tool for the same data

Why does the spec separate them at all?

So generic host applications like Claude Desktop can present a coherent UI:

  • Tools → shown as "things Claude can do during conversation"
  • Resources → shown as "data you can browse and load into context"

That's it. It's a labeling convention for UI purposes, not a technical enforcement. The spec was designed to serve Claude Desktop's sidebar UI as much as it was designed for programmatic agents.

Known criticism of the MCP spec: The three primitives overlap heavily. The Tool vs Resource distinction only makes sense in the context of a graphical host application with separate UI panels. For agent developers writing code, the distinction adds complexity without adding capability.
Practical rule for agent development: Use tools for everything. Skip resources entirely unless you are building a host application with a UI where humans browse and select context. Many production MCP servers expose everything as tools. The spec supports resources — you don't have to use them.

Prompt Templates — What They Really Are

The best mental model: prompt templates are the MCP server's way of saying "for best results, use these prompt prefixes when talking to the LLM."

The server author built the tools and knows best how to use them. Prompt templates package that expertise so every client gets consistent, optimized prompts — instead of 5 different teams writing 5 slightly different versions.

What the server author is communicating

# Server author is saying:
# "I know this domain. Here's the prompt that gets the best results
#  from the LLM when using my tools. Use this instead of guessing."

@mcp.prompt()
def stock_analysis_prompt(ticker: str, price: str) -> str:
    """Standard prompt for analyzing a stock given ticker and current price."""
    return f"""You are a financial analyst. Analyze {ticker} at ${price}.
Consider: recent earnings, market trends, and risk factors.
Give a concise BUY / HOLD / SELL recommendation with 2-3 bullet point reasons."""

Why this matters

BenefitExplanation
ConsistencyEvery client gets the same prompt — not 5 teams writing 5 slightly wrong versions
Expertise stays with the serverThe person who built the tools knows what context the LLM needs to use them well
Single update pointPrompt needs tuning? Update server once — all clients benefit immediately

The spec-level details

  • role and content are fixed by the MCP spec — you cannot add custom fields to the response
  • Valid roles: "user" or "assistant" only. "system" is not an MCP role — the client decides to send the text as a system message to the LLM
  • When you return a plain str, FastMCP auto-wraps it as PromptMessage(role="user", content=TextContent(type="text", text="..."))
  • What you define: the input parameters (names, types, defaults) — FastMCP builds the schema from type hints
# FastMCP hides this — returning a plain str auto-wraps as:
PromptMessage(
    role="user",          # spec-fixed: "user" or "assistant" only
    content=TextContent(
        type="text",       # spec-fixed: "text", "image", or "resource"
        text="You are a financial analyst..."   # your content
    )
)

# Client receives this and decides where to use it:
result = await session.get_prompt("stock_analysis_prompt", {"ticker": "AAPL", "price": "271"})
text = result.messages[0].content.text
# Client then sends text as system= or user= message to the LLM API

The honest limitation

The "avoiding inconsistency" benefit only holds if clients actually use the prompt templates — which requires discipline. The client still has to hardcode which prompt name to use for which task. If a client ignores the templates and writes its own prompts, the server can't enforce anything.

Analogy: Prompt templates are like a Postman collection shipped with an API. The API author says "here are the recommended request templates — don't guess at the format, use these." Same idea for LLM interactions. Useful convention, not technical enforcement.
