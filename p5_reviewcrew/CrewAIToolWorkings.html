Docstrings — Accessible at Runtime

A docstring is the string literal immediately after a def statement. Python stores it in the function's __doc__ attribute, which is accessible at runtime:

def fetch_pr_details(pr_url: str) -> str:
    """Fetches the title, description, changed files, and diff for a GitHub PR."""
    pass

print(fetch_pr_details.__doc__)
# "Fetches the title, description, changed files, and diff for a GitHub PR."

print(fetch_pr_details.__annotations__)
# {'pr_url': <class 'str'>, 'return': <class 'str'>}
Key point: The docstring is not a comment — it is a real Python string object stored on the function. Any code can read it at any time via .__doc__.

The @tool Decorator — What It Actually Does

A decorator is a function that takes your function and returns something else — usually a wrapper object. @tool replaces your plain function with a Tool object.

# This syntax:
@tool("fetch_pr_details")
def fetch_pr_details(pr_url: str) -> str:
    """Fetches the title..."""
    return "..."

# Is exactly equivalent to:
def fetch_pr_details(pr_url: str) -> str:
    """Fetches the title..."""
    return "..."

fetch_pr_details = tool("fetch_pr_details")(fetch_pr_details)
# The name "fetch_pr_details" now points to a Tool object, not the original function

Inside CrewAI, the tool() decorator does roughly this:

class Tool:
    def __init__(self, name, func):
        self.name = name
        self.description = func.__doc__           # reads the docstring
        self.args_schema = func.__annotations__   # reads the type hints
        self._func = func                         # stores the original function

    def run(self, input):
        return self._func(input)                  # calls original function body

Execution Sequence — Step by Step

Everything below happens the moment Python imports the file — before any agent runs:

Step 1: Python executes the def statement
    → function object created in memory
    → docstring stored in function.__doc__
    → type hints stored in function.__annotations__

Step 2: @tool decorator runs immediately after
    → receives the fully-formed function object
    → reads .__doc__ and .__annotations__
    → builds a Tool object with .name, .description, .run()

Step 3: Name "fetch_pr_details" is rebound
    → now points to the Tool object, not the original function

--- program startup complete ---

Step 4 (agent run time): LLM decides to call the tool
    → CrewAI calls fetch_pr_details.run(pr_url)
    → Tool.run() calls the original function body
    → GitHub API request is made
    → result returned to the agent
Important: Steps 1–3 happen once at import time. Step 4 happens every time the agent calls the tool. The docstring is read only once — not on every tool call.

Tool Object vs Original Function

After the decorator runs, the name points to a Tool object. The original function still exists in memory — the Tool holds a reference to it — but the name no longer reaches it directly.

Calling a Tool — Direct vs Agent

# In tests — call directly via .run()
result = fetch_pr_details.run("https://github.com/owner/repo/pull/1")

# This FAILS — Tool object is not directly callable like a function
result = fetch_pr_details("https://github.com/owner/repo/pull/1")  # TypeError

# In agent context — CrewAI calls .run() automatically
# You never call it yourself; the agent decides when to use it
Common mistake: Trying to call fetch_pr_details(url) after decoration will fail. Always use .run() in tests.

How the LLM Sees the Tool

When CrewAI sends a request to the LLM, it includes a tools section built from the Tool objects. The LLM reads this to decide which tool to call and with what input:

# What CrewAI sends to the LLM (simplified):
"""
You have access to the following tools:

fetch_pr_details: Fetches the title, description, changed files, and diff
for a GitHub pull request. Input: a full GitHub PR URL like
https://github.com/owner/repo/pull/42. Returns a formatted string with
PR metadata and the full diff.

Use the tool that best fits the task at hand.
"""

The LLM responds with a tool call (tool name + input). CrewAI intercepts it, calls .run(), and feeds the result back to the LLM. This is the same ReAct loop from P1 — CrewAI just handles it automatically.

vs P1 tool_use — Same Idea, Different API

In P1 you manually wrote the tool description as a dict. CrewAI extracts it automatically from code.

Keeping Both References (if needed)

If you need to call the original function directly (e.g., in unit tests without CrewAI), save a reference before decorating:

def _fetch_pr_details(pr_url: str) -> str:
    """Fetches the title, description..."""
    return "..."

fetch_pr_details = tool("fetch_pr_details")(_fetch_pr_details)

# Now you have both:
_fetch_pr_details(pr_url)        # original function — direct call, no CrewAI needed
fetch_pr_details.run(pr_url)     # Tool object — used by agents
