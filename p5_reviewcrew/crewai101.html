CrewAI 101

Role-based multi-agent teams — P5 ReviewCrew reference  ·  2026-04-25

What is CrewAI?

CrewAI is a high-level Python framework for building teams of AI agents that collaborate on tasks. Instead of thinking in nodes and edges (like LangGraph), you think in roles, goals, and responsibilities — like a small team of specialized workers.

Each agent has a role (who it is), a goal (what it's trying to achieve), and optionally a backstory (context that shapes its behavior). Agents are assigned tasks and can use tools. A Crew coordinates them.

Install: pip install crewai crewai-tools

The Agentic AI Stack

Agent — The Worker

An Agent is an AI worker with a defined identity. The role, goal, and backstory are all passed as text to the LLM to shape its behavior.

from crewai import Agent

reviewer = Agent(
    role="Senior Code Reviewer",
    goal="Find bugs, security issues, and bad patterns in code changes",
    backstory="You are an experienced software engineer who has reviewed "
              "thousands of PRs. You are direct, precise, and constructive.",
    tools=[github_tool],          # tools this agent can use
    llm="claude-opus-4-6",          # which LLM to use
    verbose=True,                 # print reasoning steps
    allow_delegation=False,       # can it assign work to other agents?
)
ParameterWhat it doesTips
roleJob title — shapes how LLM respondsBe specific: "Senior Python Engineer" not "Developer"
goalWhat this agent is trying to achieveOne clear sentence, action-oriented
backstoryBackground context — expertise, styleHelps LLM adopt the persona consistently
toolsList of tools this agent can callScope tightly — only what this agent needs
llmWhich model to use for this agentCan give different agents different models
allow_delegationCan this agent hand off tasks to others?Set False for leaf agents; True for manager agents
verbosePrint reasoning chain (Thought/Action/Observation)Set True during development

Task — The Assignment

A Task is a specific job for an agent. It defines what needs to be done and what the output should look like.

from crewai import Task

review_task = Task(
    description="Review the following PR diff for bugs, bad patterns, "
                 "and security issues: {pr_diff}",
    expected_output="A markdown list of issues found, each with: "
                     "severity (HIGH/MEDIUM/LOW), file, line, and explanation",
    agent=reviewer,              # which agent runs this task
    context=[fetch_task],        # wait for these tasks first, use their output
    output_pydantic=ReviewResult # optional: validate output against Pydantic model
)
ParameterWhat it does
descriptionWhat the agent should do — can use {variable} placeholders filled by crew inputs
expected_outputTells the LLM what format/content the output should be
agentWhich Agent runs this task
contextList of other Tasks whose outputs are passed as context to this task
output_pydanticValidates the LLM output against a Pydantic model

Tool — What Agents Can Do

Tools let agents interact with the outside world — fetch a URL, search the web, read a file, call an API. CrewAI has built-in tools and you can write custom ones.

# Built-in tools from crewai_tools
from crewai_tools import GithubSearchTool, SerperDevTool, FileReadTool

# Custom tool — just a decorated function
from crewai import tool

Tool scoping: assign each tool only to the agent that needs it. The agent's LLM decides when and how to call the tool — you don't call it directly.

Crew — The Team

The Crew wires everything together: which agents exist, which tasks to run, and how to coordinate them.

from crewai import Crew, Process

crew = Crew(
    agents=[fetcher, reviewer, summarizer],
    tasks=[fetch_task, review_task, summary_task],
    process=Process.sequential,   # or Process.parallel or Process.hierarchical
    verbose=True
)

result = crew.kickoff(inputs={"pr_url": "https://github.com/org/repo/pull/42"})
print(result.raw)                 # final task output as string

Sequential Process

Tasks run one after another in the order they are listed. Each task's output is available to the next task via context=. This is the default and simplest mode.

Task 1: Fetch PR ──→ Task 2: Review Code ──→ Task 3: Write Summary
    [fetcher agent]            [reviewer agent]             [summarizer agent] 
crew = Crew(
    agents=[fetcher, reviewer, summarizer],
    tasks=[fetch_task, review_task, summary_task],
    process=Process.sequential
)

Parallel Process

Multiple tasks run at the same time. Use when tasks are independent — e.g., analyzing security AND performance at the same time, then combining results.

Task 1: Fetch PR ──→ Task 2a: Security Review ──┐
                        └──→ Task 2b: Style Review ──┤ ──→ Task 3: Summary
                                                         ──┘ 
crew = Crew(
    agents=[fetcher, security_agent, style_agent, summarizer],
    tasks=[fetch_task, security_task, style_task, summary_task],
    process=Process.parallel       # security_task and style_task run simultaneously
)
Parallel gotcha: tasks marked parallel run concurrently. If Task B depends on Task A's output, use context=[task_a] — CrewAI will wait for the dependency even in parallel mode.

Hierarchical Process

A manager agent (created automatically or by you) receives the goal and decides which agents to delegate tasks to and in what order. The manager uses allow_delegation=True.

Manager Agent ──→ Delegate to CodeFetcher ──→ result
│
└──────────────→ Delegate to Reviewer ──→ result
└──────────────→ Delegate to Summarizer ──→ final output 
crew = Crew(
    agents=[fetcher, reviewer, summarizer],
    tasks=[fetch_task, review_task, summary_task],
    process=Process.hierarchical,
    manager_llm="claude-opus-4-6"    # LLM used by the auto-created manager
)

Output Passing Between Tasks

In sequential mode, each task's output is automatically passed to the next. For explicit dependency, use context=.

# Task 1: fetch the PR diff
fetch_task = Task(
    description="Fetch the diff for PR: {pr_url}",
    expected_output="The raw diff text of the pull request",
    agent=fetcher
)

# Task 2: review — depends on Task 1's output
review_task = Task(
    description="Review the PR diff for bugs and issues",
    expected_output="Markdown list of issues with severity and explanation",
    agent=reviewer,
    context=[fetch_task]     # fetch_task output is injected into this task's prompt
)

context= in Tasks

When you set context=[some_task], CrewAI appends that task's output to the current task's prompt automatically. The agent sees it as additional background information.

Behind the scenes: CrewAI builds a prompt like: "Here is context from previous tasks: [output of fetch_task]. Now do: review the PR diff for bugs..."

Pydantic Output Validation

Just like P4's structured output pattern, you can validate a task's output against a Pydantic model.

from pydantic import BaseModel

class ReviewResult(BaseModel):
    issues_found: int
    severity_breakdown: dict[str, int]   # {"HIGH": 2, "MEDIUM": 3, "LOW": 1}
    summary: str
    recommendation: str                  # "APPROVE" | "REQUEST_CHANGES" | "COMMENT"

review_task = Task(
    description="Review the PR diff...",
    expected_output="Structured review result",
    agent=reviewer,
    output_pydantic=ReviewResult         # CrewAI validates and parses the output
)

result = crew.kickoff(inputs={"pr_url": url})
review = result.pydantic                 # ReviewResult instance
print(review.recommendation)            # "REQUEST_CHANGES"

Agent — What It Really Is

An Agent in CrewAI is not code that does things autonomously. It is a configuration object that shapes how an LLM responds for a specific role. The intelligence comes from the LLM — the agent configuration just builds the system prompt.

code_reviewer = Agent(
    role="Senior Code Reviewer",        # ← persona
    goal="Find bugs and bad patterns",   # ← objective
    backstory="15 years experience...",    # ← context that shapes behavior
    tools=[],                            # ← capabilities (none = pure LLM)
    llm="claude-opus-4-6"              # ← which LLM brain to use
)

When a task is assigned, CrewAI assembles this into a system prompt behind the scenes:

# What CrewAI actually sends to the LLM:
"""
You are a Senior Code Reviewer.
Your goal is: Find bugs and bad patterns.
Background: 15 years experience reviewing production code...
You have access to these tools: [none]

Complete the following task:
[task description here]
"""
Key insight: The agent is a prompt wrapper. Changing role, goal, and backstory changes how the LLM responds — same LLM, different behavior.

ReAct Loop Inside an Agent

Every CrewAI agent runs the same ReAct loop (Reasoning + Acting) that you built manually in P1. CrewAI handles it automatically.

Task assigned to agent
│
▼
LLM receives: role + goal + backstory + task + available tools
│
▼
Thought: "I need to fetch the PR diff first"
Action: fetch_pr_details("https://github.com/...")
Observation: [tool result — the diff text]
│ (repeats until LLM says "Final Answer")
▼
Final Answer: task output returned to Crew
Agent typeLoop behaviorExample
Agent with toolsLoops: Thought → tool call → Observation → repeatCodeFetcher calls GitHub API
Agent without toolsSingle pass: Thought → Final Answer directlyCodeReviewer just analyzes the text
vs P1: In P1 you wrote the ReAct loop manually with a while True and appended tool results to the messages list. CrewAI does all of that automatically — same mechanics, abstracted away.

These solve completely different problems — they are not competing frameworks.

What it solves"Which agent does which task and in what order""How does a tool advertise itself so any LLM can call it"

Simple rule: MCP is about the plug standard. CrewAI is the team that uses the plugs. You study MCP in P6.

CrewAI vs LangChain

LangChain is about chaining LLM calls and retrievals into pipelines. CrewAI is about multiple agents collaborating on a goal. They operate at different levels and are often used together.

Core ideaChain LLM calls, retrievers, parsers into pipelinesCoordinate multiple LLM agents as a team
You defineLCEL chains: prompt | llm | parserAgents, tasks, crew process
Agent conceptOptional — an agent is one node in a chainCentral — agents are the building blocks

RelationshipCrewAI can use LangChain components internally (retrievers, parsers)

How "Branching" Works in CrewAI

CrewAI sequential mode has no real branching. Every task always runs. What looks like branching is the LLM self-directing based on instructions in the task description.

How description vs role/goal/backstory differ

What it shapesWhen used
role / goal / backstoryWHO the agent is — its system prompt personaSet once when Agent is created — permanent identity
descriptionWHAT to do right now — the task instructionPer task — changes with every task assigned

The full prompt sent to the LLM combines both:

# System prompt (from Agent):
You are a Senior Code Reviewer.
Your goal is: Find bugs and bad patterns.
Background: 15 years experience...

# User message (from Task):
Your task: Review the diff.
If you find HIGH severity issues, say 'NEEDS_SECURITY_REVIEW'.
Otherwise say 'READY_TO_MERGE'.

[context from previous tasks injected here]

The "conditional task" pattern — what actually happens

review_task = Task(
    description="""Review the diff.
    If you find HIGH severity issues, say 'NEEDS_SECURITY_REVIEW'.
    Otherwise say 'READY_TO_MERGE'.""",
    agent=reviewer
)

security_task = Task(
    description="""Only act if previous review said 'NEEDS_SECURITY_REVIEW'.
    Do a deep security audit.""",
    agent=security_specialist,
    context=[review_task]
)
What it FEELS like:
review_task ──→ NEEDS_SECURITY_REVIEW? ──→ security_task
                        ↓ no
                        summary_task

What ACTUALLY happens:
review_task ──→ security_task ──→ summary_task
  (always)        (always)         (always)

security_task LLM reads context + description:
  → sees "NEEDS_SECURITY_REVIEW" → does the audit
  → sees "READY_TO_MERGE" → responds "no action needed"
Key insight: security_task always runs and costs an LLM call — even when the review passed. You are relying on the LLM to follow the conditional instruction correctly. This is not deterministic.

Real Branching — Use LangGraph

If you want a task to truly be skipped (zero LLM calls, zero cost, deterministic), you need LangGraph's conditional edges — routing done by your Python code, not the LLM.

from langgraph.graph import StateGraph, END

def route_after_review(state):
    if "NEEDS_SECURITY_REVIEW" in state["review_output"]:
        return "security_node"
    return "summary_node"    # security_node truly never runs

graph.add_conditional_edges("review_node", route_after_review)
CrewAI "branching"LangGraph conditional edges
How routing worksLLM reads instruction, self-directsPython function checks state, routes
Skipped task truly skipped?No — always runs, LLM says "nothing to do"Yes — node never executes
CostPay for every task's LLM callPay only for nodes that run
ReliabilityDepends on LLM following instructionsDeterministic — your code decides
Use whenSimple conditional behavior is enoughReal skip logic, cost control, guaranteed routing
Production pattern: Use CrewAI for team coordination inside a node, and LangGraph for the outer workflow that decides which CrewAI crew to run and when.

Hierarchical = True Non-Linear Orchestration

LangGraph and sequential CrewAI are both hardcoded workflows — you define the structure, the LLM executes within it. CrewAI hierarchical is different: the manager LLM defines the workflow at runtime.

LangGraph / LangChainCrewAI Hierarchical
Who defines the graphYou — nodes, edges, conditions hardcodedManager LLM — at runtime, based on the goal
LLM roleExecutes within your structureExecutes AND orchestrates
Workflow known upfrontYesNo — emerges from the goal
"Agentic" levelStructured automationTrue self-direction
# LangGraph — YOU draw the graph:
graph.add_edge("fetch", "review")
graph.add_conditional_edges("review", route_fn)   # your Python code routes

# CrewAI Hierarchical — Manager LLM draws the graph at runtime:
crew = Crew(
    agents=[fetcher, reviewer, writer],
    tasks=[main_task],                              # just the goal
    process=Process.hierarchical,
    manager_llm="anthropic/claude-opus-4-7"          # LLM decides the rest
)

Same Agent Called Multiple Times

In hierarchical mode, the manager can call the same agent multiple times if completing the goal requires it — looping back, getting more context, re-analyzing with new information.

Manager: "SecurityReviewer, check this diff"
SecurityReviewer: "Found SQL injection risk on line 42"

Manager: "CodeFetcher, get full context around line 42"
CodeFetcher: returns lines 35-55

Manager: "SecurityReviewer, re-analyze with this additional context"
SecurityReviewer: "Confirmed SQL injection — here's the full analysis"

Manager: "TechWriter, write the final report"
TechWriter: returns final report

SecurityReviewer was called twice, CodeFetcher was called mid-review. You never hardcoded that — the manager decided it was necessary. In sequential mode this is impossible without rewriting the graph.

Cost implication: Each manager decision is an LLM call on top of the agent's call. If the manager loops, you pay for each decision round. Hierarchical cost = N agent calls + M manager calls, where M can exceed N.

The holy grail: Getting the autonomy of hierarchical with the reliability of LangGraph. This is an active research problem in agentic AI — 2026 and beyond. Current production systems use LangGraph for known workflows and reserve hierarchical for genuinely open-ended tasks where the flow cannot be predicted upfront.
