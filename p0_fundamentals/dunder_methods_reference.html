What Dunders Are

Dunder = "double underscore." Methods like __init__, __str__, __iter__ are not called by you directly. They are called automatically by Python when you use built-in syntax like str(x), len(x), x + y, for x in y.

Dunders are how Python's data model works — they let your custom classes participate in built-in syntax just like lists, dicts, and ints do.

How Built-ins Map to Dunders

Built-in functions and operators are syntactic sugar — they look up the corresponding dunder on the object and call it.

Built-in syntaxWhat Python calls under the hood
str(x)type(x).__str__(x)
repr(x)type(x).__repr__(x)
len(x)type(x).__len__(x)
iter(x)type(x).__iter__(x)
next(it)type(it).__next__(it)
hash(x)type(x).__hash__(x)
bool(x)type(x).__bool__(x) (or __len__ if no __bool__)
x + ytype(x).__add__(x, y)
x[k]type(x).__getitem__(x, k)
k in xtype(x).__contains__(x, k)
obj(args)type(obj).__call__(obj, args)
with x as y:x.__enter__() ... x.__exit__()
for v in x:x.__iter__() then repeated __next__()
obj.attrtype(obj).__getattribute__(obj, "attr")
print(x) / f"{x}"__str__ (falls back to __repr__)
Why this indirection exists — Polymorphism

This design lets str(x), len(x), for x in y work on any object. The interpreter doesn't care what type x is — it just asks x how to do the operation. You don't write if isinstance(x, int): ... elif isinstance(x, list): .... Python dispatches automatically.

⚠️ Dunders must be on the CLASS, not the instance
class Stock:
    def __str__(self): return "stock"

s = Stock()
s.__str__ = lambda: "different"   # ❌ adding to instance — IGNORED
print(str(s))   # 'stock'  — uses the class method, not instance
Python's dispatcher looks at type(s).__str__, not s.__str__. Adding dunders to instances doesn't work.

2. String Representation

MethodWhen calledWhat it should return
__repr__(self)repr(x), debugger, list/dict display, !r in f-stringUnambiguous, ideally code that recreates the object
__str__(self)str(x), print(x), f"{x}"Human-friendly text
__format__(self, spec)f"{x:spec}", format(x, spec)Custom formatting
 
class Stock:
    def __repr__(self):
        return f"Stock({self.ticker!r}, {self.price})"   # Stock('AAPL', 150)
    def __str__(self):
        return f"{self.ticker}: ${self.price}"           # AAPL: $150

s = Stock("AAPL", 150)
print(s)         # 'AAPL: $150'      ← uses __str__
str(s)           # 'AAPL: $150'
repr(s)          # "Stock('AAPL', 150)"
[s]              # [Stock('AAPL', 150)]   ← lists show __repr__
f"{s!r}"         # "Stock('AAPL', 150)"   ← !r forces __repr__
__str__ vs __repr__ rule
If __str__ isn't defined, Python falls back to __repr__. So if you only define one, define __repr__ — it covers both cases.

3. Iteration

MethodWhen calledReturns
__iter__(self)iter(x), for x in objAn iterator
__next__(self)next(it)Next value, or raises StopIteration
 
class CountDown:
    def __init__(self, n): self.n = n
    def __iter__(self): return self
    def __next__(self):
        if self.n <= 0: raise StopIteration
        self.n -= 1
        return self.n + 1

for x in CountDown(3): print(x)   # 3 2 1
A for loop is just iter() + next() + StopIteration
# This loop:
for x in [1, 2, 3]:
    print(x)

# Is equivalent to:
it = iter([1, 2, 3])
while True:
    try:
        x = next(it)
        print(x)
    except StopIteration:
        break

4. Container Behavior

MethodWhen called
__len__(self)len(x)
__getitem__(self, key)x[key]
__setitem__(self, key, value)x[key] = value
__delitem__(self, key)del x[key]
__contains__(self, item)item in x
 
class Portfolio:
    def __init__(self): self.holdings = {}
    def __len__(self): return len(self.holdings)
    def __getitem__(self, ticker): return self.holdings[ticker]
    def __setitem__(self, ticker, qty): self.holdings[ticker] = qty
    def __contains__(self, ticker): return ticker in self.holdings

p = Portfolio()
p["AAPL"] = 100         # __setitem__
print(len(p))           # __len__ → 1
print("AAPL" in p)      # __contains__ → True
print(p["AAPL"])        # __getitem__ → 100

5. Comparison and Hashing

🚨 Define __hash__ when you define __eq__
If you define __eq__, Python automatically sets __hash__ = None, making the object unhashable. If you want it to remain usable as a dict key, define both — and make sure equal objects have equal hashes.

7. Context Managers (the with statement)

MethodWhen called
__enter__(self)with obj as x: — entering the block. Return value bound to x.
__exit__(self, exc_type, exc_val, tb)Leaving the block — runs even if exception raised
 
class FileLogger:
    def __enter__(self):
        self.f = open("log.txt", "a")
        return self.f
    def __exit__(self, exc_type, exc_val, tb):
        self.f.close()

with FileLogger() as f:
    f.write("hello")
# f auto-closed even if write() raised

9. Callable

class Multiplier:
    def __init__(self, factor): self.factor = factor
    def __call__(self, x): return x * self.factor

triple = Multiplier(3)
triple(10)      # 30 — instance is callable like a function

CrewAI's BaseTool._run(), agent wrappers, retry decorators all use this pattern.

The if __name__ == "__main__" Idiom

__name__ is "__main__" ONLY when you launch that file directly. When the file is imported by another file, __name__ becomes the module's name (filename without .py).

Demo

# tools.py
print(f"tools.py loaded — __name__ is: {__name__}")

def get_price(ticker):
    return 150.0

if __name__ == "__main__":
    print("Running tools.py as a script — running tests")
    print(get_price("AAPL"))
# client.py
from tools import get_price
print(f"client.py — __name__ is: {__name__}")
print(get_price("AAPL"))

Run tools.py directly:

$ python tools.py
tools.py loaded — __name__ is: __main__
Running tools.py as a script — running tests
150.0

Run client.py (which imports tools.py):

$ python client.py
tools.py loaded — __name__ is: tools          ← imported, NOT __main__
client.py — __name__ is: __main__              ← the launched file IS __main__
150.0
ScenarioFile launched__name__ in launched file__name__ in imported files
python tools.pytools.py"__main__"(none)
python client.pyclient.py"__main__"tools.py: "tools"
python server.pyserver.py"__main__"tools.py: "tools"
Why this idiom matters
The pattern lets a file be two things at once:
  • A library when imported (only definitions loaded)
  • A script when run directly (also runs the test/demo block)
Without it, every import would also run the test code at the bottom.

Real example — p6_mcp/server.py

mcp = FastMCP("StockSage Analysis Server", host="0.0.0.0", port=8002)

@mcp.tool()
def stock_price_data(ticker, period="3mo"):
    return str(get_price_data(ticker, period))

if __name__ == "__main__":
    mcp.run(transport="streamable-http")
  • When you run python server.py → __name__ == "__main__" → server starts
  • If something imported server.py → __name__ == "server" → server does NOT start. The importer just gets the mcp object and tool definitions.

Why Python Doesn't Support Method Overloading

You cannot define the same method twice with different signatures — the second definition overwrites the first.

class Stock:
    def __init__(self, ticker: str):
        ...
    def __init__(self, ticker: str, price: float):    # OVERWRITES the first
        ...

# Only the second __init__ exists
s = Stock("AAPL")          # ❌ TypeError: missing 'price'
s = Stock("AAPL", 150.0)   # ✅ works

Why: Python is dynamically typed — no compile-time type info to dispatch on. Every name in a class is a regular dict entry; defining the same name twice just overwrites.

The 4 Pythonic alternatives

1. Default arguments (most common)

class Stock:
    def __init__(self, ticker: str, price: float = 0.0, signal: str = "HOLD"):
        ...

Stock("AAPL")
Stock("AAPL", 150.0)
Stock("AAPL", signal="BUY")

2. Alternative constructors via @classmethod (most Pythonic)

class Stock:
    def __init__(self, ticker, price):
        self.ticker, self.price = ticker, price

    @classmethod
    def from_dict(cls, data):
        return cls(ticker=data["ticker"], price=data["price"])

    @classmethod
    def from_string(cls, s):
        ticker, price = s.split(":")
        return cls(ticker, float(price))

Stock("AAPL", 150)
Stock.from_dict({"ticker": "AAPL", "price": 150})
Stock.from_string("AAPL:150")

You see this pattern everywhere: dict.fromkeys, datetime.fromisoformat, Path.cwd, Pydantic's Model.model_validate.

3. Type checking inside one method (used sparingly)

def __init__(self, value):
    if isinstance(value, str): ...
    elif isinstance(value, dict): ...
    elif isinstance(value, tuple): ...

4. functools.singledispatchmethod (true type-based dispatch)

from functools import singledispatchmethod

class Stock:
    @singledispatchmethod
    def update(self, value): raise NotImplementedError

    @update.register
    def _(self, value: str): self.ticker = value
    @update.register
    def _(self, value: float): self.price = value
typing.overload — for type checkers only, not runtime
from typing import overload

class Stock:
    @overload
    def __init__(self, ticker: str) -> None: ...
    @overload
    def __init__(self, ticker: str, price: float) -> None: ...

    def __init__(self, ticker, price=0.0):    # actual implementation
        ...
These @overload decorations are only documentation for type checkers (mypy, pyright). At runtime, only the last __init__ exists. Useful for libraries with complex signatures.
