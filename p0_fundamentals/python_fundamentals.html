Lists — Operations & Tricks

Creating

tickers = ["AAPL", "GOOGL", "MSFT"]
empty = []
from_range = list(range(5))     # [0, 1, 2, 3, 4]
repeated = [0] * 3              # [0, 0, 0]

Indexing & Slicing

tickers[0]       # "AAPL"          first
tickers[-1]      # "MSFT"          last (negative from end)
tickers[0:2]     # ["AAPL", "GOOGL"]   stop is exclusive
tickers[1:]      # ["GOOGL", "MSFT"]
tickers[:2]      # ["AAPL", "GOOGL"]
tickers[::-1]    # reversed
tickers[::2]     # every other

Adding

tickers.append("AMD")              # one item at end
tickers.insert(0, "TSLA")          # at index 0
tickers.extend(["NVDA", "META"])   # multiple

a + b                              # NEW list (doesn't mutate)
⚠️ append vs extend
tickers.append(["X", "Y"])   # → [..., ["X", "Y"]]   nested!
tickers.extend(["X", "Y"])   # → [..., "X", "Y"]    flat

Removing

tickers.remove("AAPL")    # by VALUE (first occurrence)
del tickers[0]            # by INDEX
tickers.pop()             # remove + return last
tickers.pop(0)            # remove + return at index
tickers.clear()           # empties

Searching

"AAPL" in tickers         # True / False
tickers.index("AAPL")     # int index, ValueError if missing
tickers.count("AAPL")     # how many times

Sorting

tickers.sort()                       # in-place
sorted(tickers)                      # NEW list
sorted(tickers, reverse=True)
sorted(tickers, key=len)             # by length

Combining / Splitting

[1, 2] + [3, 4]            # [1, 2, 3, 4]
[0] * 3                    # [0, 0, 0]
",".join(tickers)          # "AAPL,GOOGL,MSFT"
"AAPL,GOOGL".split(",")    # ["AAPL", "GOOGL"]

Aggregation

sum([1, 2, 3])      # 6
min(tickers)        # alphabetical min
max([1, 2, 3])      # 3
any([0, 0, 1])      # True
all([1, 1, 0])      # False

List Comprehensions

Pattern: [expression for item in iterable if condition]

tickers = ["AAPL", "GOOGL", "MSFT", "AMD", "TSLA"]

# Filter
a_only = [t for t in tickers if t.startswith("A")]

# Transform
lower = [t.lower() for t in tickers]

# Both
short_lower = [t.lower() for t in tickers if len(t) <= 4]

# Equivalent to this loop
result = []
for t in tickers:
    if t.startswith("A"):
        result.append(t)
Why use comprehensions
Faster than equivalent loop (less bytecode, no .append method lookup), and more readable for transform/filter operations. Idiomatic Python.

Nested Lists (Lists of Lists)

matrix = [
    ["AAPL", 150.0, "BUY"],
    ["GOOGL", 2800.0, "HOLD"],
]

matrix[0]        # ["AAPL", 150.0, "BUY"]
matrix[0][1]     # 150.0
matrix[-1][-1]   # "HOLD"

# Iterate with unpacking
for ticker, price, signal in matrix:
    print(f"{ticker}: ${price} → {signal}")
🚨 The classic [[0]*N]*M trap
grid = [[0] * 3] * 3     # ❌ all 3 rows are the SAME inner list
grid[0][0] = 9
print(grid)
# [[9, 0, 0], [9, 0, 0], [9, 0, 0]]   ← all rows mutated!

The outer * 3 creates 3 references to the same inner list. id() proves it:

grid = [[0] * 3] * 3
print(id(grid[0]), id(grid[1]), id(grid[2]))
# 4399876928 4399876928 4399876928   ← SAME id for all three rows
Fix — use a comprehension to create fresh inner lists: 
grid = [[0] * 3 for _ in range(3)]   # ✅ correct
grid[0][0] = 9
# [[9, 0, 0], [0, 0, 0], [0, 0, 0]]

Dicts — Operations & Tricks

Creating

prices = {"AAPL": 150.0, "GOOGL": 2800.0}
empty = {}
from_pairs = dict([("AAPL", 150), ("GOOGL", 2800)])

Lookup

prices["AAPL"]                    # raises KeyError if missing
prices.get("AAPL")                # returns None if missing
prices.get("UNKNOWN", "N/A")      # default value if missing
"AAPL" in prices                  # True / False

Add / Update / Delete

prices["MSFT"] = 300.0           # add
prices["AAPL"] = 175.0           # update (same syntax)

del prices["GOOGL"]              # delete by key
prices.pop("MSFT", None)         # delete + return value (or None default)
prices.clear()                   # empty

Iterating

for key in prices:                       # keys
    ...
for key in prices.keys():                # explicit keys
    ...
for value in prices.values():            # values
    ...
for key, value in prices.items():        # pairs (most common)
    ...

Dict Keys — What's Allowed

Keys must be hashable (immutable).

TypeHashable?Can be a key?
int, float✅✅ {150: "AAPL"}
str✅✅ {"AAPL": 150}
tuple (of hashables)✅✅ {("AAPL", "Q1"): 150}
bool, None✅✅
list❌❌ TypeError
dict❌❌ TypeError
set❌❌ (use frozenset)
⚠️ 1, 1.0, and True share a slot
weird = {1: "int", 1.0: "float", True: "bool"}
print(weird)
# {1: 'bool'}    — only one entry!
Why? 1 == 1.0 == True AND they hash to the same value. Don't mix numeric types as keys.

Tuple keys for compound lookups

revenue = {
    ("AAPL", "Q1"): 124_000_000_000,
    ("AAPL", "Q2"): 96_000_000_000,
    ("MSFT", "Q1"): 65_000_000_000,
}
revenue[("AAPL", "Q1")]   # 124000000000

Dict Values — Anything Goes

Values are unconstrained. Different types within the same dict are fine if they represent structured data.

# Mixed value types — totally legal
chaos = {
    "ticker": "AAPL",                       # string
    "price": 150.0,                         # float
    "history": [148, 149, 150],             # list
    "metadata": {"sector": "tech"},         # nested dict
    "callback": lambda x: x * 2,            # function
    "buyer": None,                          # None
}
Mixed values are appropriate when
  • The dict represents one structured record (like JSON, MCP tool input)
  • Keys have fixed meanings — not interchangeable
When NOT to mix: If you're using the dict as a lookup table (homogeneous keys → values), mixed types make consumers type-check everything. Switch to Pydantic / dataclass.

Dict Merging with the | Operator

Python 3.9+ — merge dicts with |. Right side wins on key collision.

defaults = {"period": "3mo", "lang": "en"}
overrides = {"period": "1y"}

merged = defaults | overrides
# {'period': '1y', 'lang': 'en'}    — overrides won

# In-place
config = defaults.copy()
config |= overrides

# Pre-3.9 way (still works, more verbose)
merged = {**defaults, **overrides}

The full rule for a | b

Key locationResult
Only in aKept from a
Only in bAdded from b
In bothValue from b wins (right side)

The | operator is overloaded

Operand typesWhat | does
bool, boolLogical OR (but prefer or — it short-circuits)
int, intBitwise OR (5 | 3 = 7)
set, setSet union
dict, dictMerge with right side winning
Custom classWhatever __or__ defines (LangGraph edges)

Dict Comprehensions

Pattern: {key_expr: value_expr for k, v in dict.items() if cond}

prices = {"AAPL": 150, "GOOGL": 2800, "MSFT": 300, "AMD": 100}

# Filter — keep only expensive
expensive = {t: p for t, p in prices.items() if p > 200}
# {'GOOGL': 2800, 'MSFT': 300}

# Transform values
in_thousands = {t: p / 1000 for t, p in prices.items()}

# Build dict from a list
tickers = ["AAPL", "GOOGL", "MSFT"]
init_zero = {t: 0.0 for t in tickers}

# Invert keys/values
inverted = {v: k for k, v in prices.items()}
Critical: use .items() for unpacking
# Wrong — iterates KEYS only
{t: ??? for t in prices if ???}

# Right — iterates (key, value) pairs
{t: p for t, p in prices.items() if p > 200}

Nested Dicts

portfolio = {
    "AAPL": {"price": 270.0, "pe": 34.3, "signal": "BUY"},
    "GOOGL": {"price": 2800.0, "pe": 28.5, "signal": "HOLD"},

portfolio["AAPL"]["price"]                 # 270.0   direct
portfolio.get("AMD", {}).get("signal")     # None    safe chain — no KeyError

Multiple Values per Key

Dict keys are unique — assigning to an existing key overwrites. To "store multiple per key," make the value a list.

from collections import defaultdict

trades = defaultdict(list)        # auto-creates [] if key missing
trades["AAPL"].append(150.0)
trades["AAPL"].append(175.0)
trades["MSFT"].append(300.0)
# defaultdict(list, {'AAPL': [150.0, 175.0], 'MSFT': [300.0]})

Counter for "how many of each"

from collections import Counter

trades_ticker = ["AAPL", "MSFT", "AAPL", "GOOGL", "AAPL"]
counts = Counter(trades_ticker)
# Counter({'AAPL': 3, 'MSFT': 1, 'GOOGL': 1})

counts.most_common(2)            # [('AAPL', 3), ('MSFT', 1)]
PART 2 — Mutability

Mutable vs Immutable — The Foundation

Mutable = you can change the object's contents without creating a new object.
Immutable = once created, contents are sealed. "Modifying" actually creates a new object.

MutableImmutable
Can change contents in-place?✅ Yes❌ No
Can change size?✅ Yes❌ No
Modification creates new object?❌ No — same object, new contents✅ Yes — entirely new object

See it with id()

Mutable — same object, contents change

nums = [1, 2, 3]
print(id(nums))     # 4399876928

nums.append(4)      # modify in-place
print(nums)         # [1, 2, 3, 4]
print(id(nums))     # 4399876928   ← SAME id — same object, new contents

nums[0] = 99        # change one element — also in-place
print(id(nums))     # 4399876928   ← STILL same id

Immutable — modification creates a brand new object

s = "hello"
print(id(s))        # 4399890432

s = s + "!"         # looks like modification, but...
print(s)            # 'hello!'
print(id(s))        # 4399897856   ← DIFFERENT id — entirely new string

TypeExampleWhy immutable

str"hellothere"Sealed character sequence
tuple(1, 2, 3)Sealed sequence

Every literal you type — "hellothere", 12.0, True, (1, 2) — is an immutable object.

Rebinding vs Mutating

This is the subtle part that trips people up:

x = "hello"             # x points to "hello"
x = "world"             # x now points to "world" — original "hello" unchanged

nums = [1, 2, 3]        # nums points to a list
nums = [4, 5, 6]        # rebound — original [1, 2, 3] gets garbage-collected
nums.append(7)          # mutated — same list as above, now contains [4, 5, 6, 7]
OperationWhat it does
x = somethingRebind — point variable at a different object
x.method() (on mutable)Mutate — change the object x already points to

Why this matters — function argument behavior

def modify(items):
    items.append(99)        # mutates the caller's list

nums = [1, 2, 3]
modify(nums)
print(nums)                  # [1, 2, 3, 99]  ← caller's list mutated!

def modify_str(s):
    s += "!"                 # rebinds local variable — caller's s unchanged

text = "hello"
modify_str(text)
print(text)                  # 'hello'  — unchanged

Mutable arguments can be modified by the function and the caller sees the change. Immutable arguments can't be — modifications inside the function only affect the local binding.

Mutability Gotchas

🚨 1. Assignment is NOT a copy
a = [1, 2, 3]
b = a               # b and a point to the SAME list
b.append(4)
print(a)            # [1, 2, 3, 4]  — mutated through b
Fix: b = a.copy() · b = a[:] · b = list(a). For nested: copy.deepcopy(a).
🚨 2. Modifying a dict during iteration
for k in prices:
    if prices[k] < 200:
        del prices[k]      # RuntimeError: dictionary changed size during iteration
Fix: iterate over a copy: for k in list(prices.keys()):
🚨 3. Nested list multiplication trap
grid = [[0] * 3] * 3     # all rows share the SAME inner list
grid[0][0] = 9
# [[9, 0, 0], [9, 0, 0], [9, 0, 0]]
Fix: [[0] * 3 for _ in range(3)]
🚨 4. Mutable default arguments
def add_trade(ticker, history=[]):     # ❌ default created ONCE at function def
    history.append(ticker)
    return history

add_trade("AAPL")     # ['AAPL']
add_trade("MSFT")     # ['AAPL', 'MSFT']  — surprised?
Fix: use None sentinel: 
def add_trade(ticker, history=None):
    if history is None:
        history = []
    history.append(ticker)
    return history
🚨 5. Dict key collisions (numeric)
{1: "a", 1.0: "b", True: "c"}    # → {1: 'c'}    only ONE entry
Don't mix numeric types as keys.
🚨 6. .append() returns None
tickers = ["AAPL"].append("MSFT")      # ❌ tickers is None!
Fix: mutate then read separately. Same for .sort(), .extend(), .clear(), .update().
PART 3 — Functional Programming

Lambdas — Anonymous Functions

A lambda is a tiny one-line function with no name.

def square(x):
    return x * x

square_lambda = lambda x: x * x

square(5)         # 25
square_lambda(5)  # 25

Syntax

lambda <args>: <single expression>

# Examples
lambda x: x * 2
lambda a, b: a + b
lambda name, prefix="Hello": f"{prefix}, {name}"
lambda x: x if x >= 0 else -x      # ternary OK
When to use lambda vs def
  • Use lambda when passing a tiny one-shot function as an argument: sorted(x, key=lambda t: t.price)
  • Use def when the function has a name worth referring to, has multiple lines, or is used in more than one place

Lambda Limitations

AllowedNot allowed
One expressionStatements (assign, loop, print, return)
Conditional expression (x if cond else y)Multi-line bodies
Default argumentsAnnotations easily

map(func, iterable)

Apply a function to every item. Returns an iterator (lazy), not a list.

prices = [100, 200, 300]

doubled_iter = map(lambda x: x * 2, prices)
print(doubled_iter)         # <map object at 0x...>   ← iterator, not list
print(list(doubled_iter))   # [200, 400, 600]         ← consume to see

Map over multiple iterables

a = [1, 2, 3]
b = [10, 20, 30]
list(map(lambda x, y: x + y, a, b))   # [11, 22, 33]
"iterable" parameter accepts any iterable, not just list
The signature map(func, iterable) uses "iterable" because map works with anything you can loop over: list, tuple, set, str, dict, range, generator, file. A list IS an iterable — but so are many other types.

filter(func, iterable)

prices = [100, 250, 175, 300, 50]
expensive = list(filter(lambda x: x > 200, prices))
print(expensive)   # [250, 300]

# filter(None, ...) — drop falsy values
mixed = [1, 0, "hello", "", None, [1], []]
list(filter(None, mixed))   # [1, 'hello', [1]]

Map / Zip with Unequal-Length Iterables

map and zip stop at the SHORTEST iterable — silently dropping extras.

a = [1, 2, 3]
b = [10, 20, 30, 40]
list(map(lambda x, y: x + y, a, b))   # [11, 22, 33]   ← 40 silently dropped
list(zip(a, b))                        # [(1, 10), (2, 20), (3, 30)]
You want…Use
Silently truncate to shortestmap(func, a, b) or zip(a, b)
Error if lengths differzip(a, b, strict=True) (Python 3.10+)
Pad shorter to longeritertools.zip_longest(a, b, fillvalue=...)
For agentic code — prefer strict=True
Silent data loss in agent dispatch (e.g., zipping tool names with inputs and dropping one) is a nightmare to debug.

sorted(iterable, key=…, reverse=…)

Returns a NEW sorted list. Doesn't mutate the input.

nums = [3, 1, 4, 1, 5, 9, 2, 6]
sorted(nums)                  # [1, 1, 2, 3, 4, 5, 6, 9]
sorted(nums, reverse=True)    # [9, 6, 5, 4, 3, 2, 1, 1]

# Sort by length using key
sorted(["MSFT", "AAPL", "GOOGL"], key=len)    # ['MSFT', 'AAPL', 'GOOGL']

list.sort() vs sorted()

FunctionMutates?Returns?
x.sort()Yes — modifies xNone

🚨 list.sort() returns None
tickers = ["MSFT", "AAPL"].sort()    # ❌ tickers is None!

# ✅ Either:
sorted_tickers = sorted(["MSFT", "AAPL"])
# Or:
tickers = ["MSFT", "AAPL"]
tickers.sort()

Sorting Dicts and Objects

Sort list of tuples by element index

prices = [("AAPL", 150), ("GOOGL", 2800), ("MSFT", 300)]
sorted(prices, key=lambda x: x[1])

# Same with operator.itemgetter (faster, more idiomatic)
from operator import itemgetter
sorted(prices, key=itemgetter(1))

Sort list of dicts by key

portfolio = [
    {"ticker": "AAPL", "price": 150, "pe": 34},
    {"ticker": "GOOGL", "price": 2800, "pe": 28},
]

sorted(portfolio, key=lambda x: x["pe"])

Sort list of objects by attribute

from operator import attrgetter
sorted(stocks, key=attrgetter("price"))

Multi-Key Sorting

Return a tuple from the key function. Python compares tuples lexicographically.

sorted(portfolio, key=lambda x: (x["pe"], x["price"]))

Mixed asc/desc — negate numeric fields

sorted(portfolio, key=lambda x: (x["pe"], -x["price"]))
Stable sort matters
Python's sorted is stable — items with equal keys keep original order. So sorting twice (last-key-first) gives correct multi-key results.

reduce()

from functools import reduce

nums = [1, 2, 3, 4, 5]
reduce(lambda acc, x: acc + x, nums)         # 15
reduce(lambda acc, x: acc * x, nums)         # 120
reduce(lambda acc, x: acc + x, nums, 100)    # 115 (with initial)
Use sum(), max(), min(), any(), all() instead when possible — clearer.

PART 4 — Iteration Protocol

Iterable vs Iterator — The Distinction

Every iterator IS an iterable — but not every iterable is an iterator.

┌─────────────────────────────────┐
│ ITERABLE (has __iter__)         │
│   Examples: list, tuple, str,   │
│   set, dict, range              │
│                                 │
│   ┌───────────────────────────┐ │
│   │ ITERATOR                  │ │
│   │ (also has __next__)       │ │
│   │                           │ │
│   │ - has internal bookmark   │ │
│   │ - one-shot only           │ │
│   │                           │ │
│   │ Examples: map, filter,    │ │
│   │ generator, file, iter([]) │ │
│   └───────────────────────────┘ │
└─────────────────────────────────┘
IterableIterator
Has __iter__✅✅ (returns itself)
Has __next__❌✅
Works in for x in ...✅✅
Works with next(...)❌✅
Reusable✅❌ (one-shot)
Exampleslist, str, dict, range, set, tuplemap, filter, generator, file, iter(anything)

Test which is which

from collections.abc import Iterable, Iterator

isinstance([1, 2, 3], Iterable)         # True
isinstance([1, 2, 3], Iterator)         # False  ← list is NOT an iterator

it = iter([1, 2, 3])
isinstance(it, Iterator)                # True   ← now it's both

m = map(lambda x: x*2, [1, 2, 3])
isinstance(m, Iterator)                 # True

How for-loops Actually Work

A for loop is just iter() + next() + catch StopIteration:

# This loop:
for x in [1, 2, 3]:
    print(x)

# Is equivalent to:
it = iter([1, 2, 3])
while True:
    try:
        x = next(it)
        print(x)
    except StopIteration:
        break

The mental model — iterator as a bookmark

Stream:  [1, 2, 3, 4, 5]
              ↑
           bookmark (advances on each next())

After 3 calls to next():
Stream:  [1, 2, 3, 4, 5]
                    ↑
                 bookmark

When the bookmark falls off the end → StopIteration

Why Iterators Exist (Laziness)

Iterators don't hold all values in memory at once.

# List: ALL 1 billion ints exist in RAM at once → ~8GB
nums = [x * 2 for x in range(1_000_000_000)]   # crashes

# Iterator: values produced on demand → constant memory
nums = (x * 2 for x in range(1_000_000_000))   # ✅ instant, ~zero memory
for n in nums:
    if n > 100: break    # only computed first ~50 values
⚠️ Iterators are exhausted after one pass
m = map(lambda x: x * 2, [1, 2, 3])
list(m)        # [2, 4, 6]
list(m)        # []  ← already consumed
Use a list (or call iter(original) again) if you need to iterate twice.

range — A Lazy Sequence

range is its own type — iterable AND reusable AND lazy. Best of all worlds.

r = range(5)
print(type(r))    # <class 'range'>
print(r)          # range(0, 5)    ← doesn't show contents
print(list(r))    # [0, 1, 2, 3, 4]

# Reusable (unlike iterators)
for x in range(3): print(x)   # 0 1 2
for x in range(3): print(x)   # 0 1 2  ← works again

# Supports indexing (unlike generators)
r[100]      # 100
r[10:20]    # slice → another range object
len(r)      # length without iterating
500 in range(1000)    # O(1) — uses arithmetic, not iteration

range stores just 3 numbers (start, stop, step) and computes values on demand.

Generators — yield

A function with yield is a generator function. Calling it returns a generator (an iterator).

def count_up_to(n):
    i = 0
    while i < n:
        yield i        # pause here, return value, resume on next()
        i += 1

gen = count_up_to(3)
print(next(gen))   # 0
print(next(gen))   # 1
print(next(gen))   # 2
print(next(gen))   # StopIteration

for x in count_up_to(5):
    print(x)       # 0 1 2 3 4

Generator expression — like comprehension but lazy

# List comprehension — eagerly builds full list
squares = [x**2 for x in range(10)]    # [0, 1, 4, 9, ..., 81]

# Generator expression — lazy iterator (parens instead of brackets)
squares = (x**2 for x in range(10))    # <generator object>
print(list(squares))  # only now materialized

Why print(list) Shows Contents but print(map) Doesn't

print() calls __repr__ on the object. Different types implement it differently.

prices = [100, 200, 300]
print(prices)                # [100, 200, 300]
                             # list.__repr__ walks elements — safe (in memory already)

doubled = map(lambda x: x * 2, prices)
print(doubled)               # <map object at 0x10c8b3940>
                             # map.__repr__ shows placeholder
                             # — showing values would CONSUME the iterator!
Iterator __repr__ design rationale
Iterators don't show contents in repr — that would require consuming them. Imagine an infinite iterator: itertools.count(1) would loop forever trying to print. The <...> placeholder is the only safe answer.

Same for all iterators

print(map(lambda x: x, [1, 2, 3]))           # <map object at 0x...>
print(filter(lambda x: x, [1, 2, 3]))        # <filter object at 0x...>
print(zip([1], [2]))                          # <zip object at 0x...>
print(enumerate([1, 2, 3]))                  # <enumerate object at 0x...>
print((x for x in [1, 2, 3]))                # <generator object at 0x...>

To see contents — wrap with list(...) to materialize (this exhausts the iterator).

The 3 Method Types — Instance, Class, Static

class Stock:
    market = "NYSE"                    # class attribute (shared)

    def __init__(self, ticker, price):
        self.ticker = ticker
        self.price = price

    def describe(self):

    @classmethod

@staticmethod

Just a regular function namespaced under the class — receives no implicit first argument.

class TempConverter:
    @staticmethod
    def f_to_c(f): return (f - 32) * 5 / 9
    @staticmethod
    def c_to_f(c): return c * 9 / 5 + 32

TempConverter.f_to_c(212)    # 100.0

Use when the function is logically related to the class but doesn't need any instance or class state. Pure organizational decoration.

@classmethod (and the Factory Pattern)

Receives the class itself (cls) as first argument. Can create new instances with cls(...) — automatically subclass-aware.

class Stock:
    def __init__(self, ticker, price):
        self.ticker = ticker
        self.price = price

    @classmethod
    def from_dict(cls, data):
        return cls(ticker=data["ticker"], price=data["price"])

    @classmethod
    def from_csv(cls, row):
        ticker, price = row.split(",")
        return cls(ticker, float(price))

    @classmethod
    def from_yfinance(cls, ticker):
        info = yfinance.Ticker(ticker).info
        return cls(ticker, info["currentPrice"])

# Multiple "constructors", clearly named
Stock("AAPL", 150)
Stock.from_dict({"ticker": "AAPL", "price": 150})
Stock.from_csv("AAPL,150")
Stock.from_yfinance("AAPL")

You see this pattern everywhere:

  • dict.fromkeys(...)
  • datetime.fromtimestamp(...), datetime.fromisoformat(...)
  • Path.cwd(), Path.home()
  • Pydantic Model.model_validate(dict), Model.model_validate_json(str)
  • LangChain HumanMessage.from_dict(...)
Why cls (vs hardcoding the class)
class Stock:
    @staticmethod
    def from_dict_static(data):
        return Stock(...)        # ❌ HARDCODED Stock

    @classmethod
    def from_dict_cls(cls, data):
        return cls(...)          # ✅ uses whatever class called this

class TechStock(Stock): pass

TechStock.from_dict_static({...})    # → Stock instance (WRONG)
TechStock.from_dict_cls({...})       # → TechStock instance (CORRECT)
Class methods are subclass-aware automatically.

self is Just a Parameter Name

self is NOT a Python keyword. It's just a regular parameter name — convention only.

class Stock:
    def __init__(me, ticker):       # 'me' instead of 'self' — works
        me.ticker = ticker

    def describe(this):              # 'this' (C++ style) — also works
        return this.ticker

s = Stock("AAPL")
print(s.describe())                  # 'AAPL'

Python doesn't care about the name. It just passes the instance as the first positional argument, whatever you name that parameter.

Why self gets passed automatically — the descriptor protocol

When you access s.describe, Python doesn't return the raw function — it returns a bound method (instance pre-wired). This is the descriptor protocol at work. Functions defined in a class body are descriptors; their __get__ method returns a bound version when accessed via an instance.

print(s.describe)         # <bound method Stock.describe of <Stock object>>
                          # ← BOUND — self already wired in
print(Stock.describe)     # <function Stock.describe at 0x...>
                          # ← UNBOUND — must pass self explicitly

s.describe()              # auto-passes s
Stock.describe(s)         # explicit pass — same result

By convention, ALWAYS use self

PEP 8 and every Python style guide say: self for instance methods, cls for class methods. Renaming works but breaks every reader's expectation.

Why Python Doesn't Support Method Overloading

You cannot define the same method twice with different signatures — the second definition overwrites the first.

class Stock:
    def __init__(self, ticker: str):
        ...
    def __init__(self, ticker: str, price: float):    # OVERWRITES
        ...

# Only the second __init__ exists
s = Stock("AAPL")          # ❌ TypeError: missing 'price'

Why: Python is dynamically typed — no compile-time type info to dispatch on. Every name in a class is a regular dict entry; defining the same name twice just overwrites.

The 4 Pythonic alternatives

1. Default arguments (most common)

def __init__(self, ticker, price=0.0, signal="HOLD"):
    ...

2. @classmethod factories (most Pythonic)

Stock.from_dict({...})
Stock.from_string("AAPL:150")
Stock.from_yfinance("AAPL")

3. Type checking inside one method (rarely)

def __init__(self, value):
    if isinstance(value, str): ...
    elif isinstance(value, dict): ...

4. functools.singledispatchmethod

from functools import singledispatchmethod

class Stock:
    @singledispatchmethod
    def update(self, value): raise NotImplementedError
    @update.register
    def _(self, value: str): self.ticker = value
    @update.register
    def _(self, value: float): self.price = value

PART 6 — Dunder Methods (Python's Data Model)

What Dunders Are

Dunder = "double underscore." Methods like __init__, __str__, __iter__ are not called by you directly. They are called automatically by Python when you use built-in syntax like str(x), len(x), x + y, for x in y.

Dunders are how Python's data model works — they let your custom classes participate in built-in syntax just like lists, dicts, and ints do.

Dunders Must Live on the Class

Dunders are looked up via type(x).__method__, NOT x.__method__. This means adding them to an instance after creation does NOT affect built-in syntax.

class Stock:
    def __str__(self): return "stock"

s = Stock()

# ❌ Adding to instance — IGNORED for built-in syntax
s.__str__ = lambda: "instance dunder"
print(str(s))                # 'stock' — class method still wins

# ✅ Adding to class — TAKES EFFECT for ALL instances
Stock.__str__ = lambda self: "patched class dunder"
print(str(s))                # 'patched class dunder'
Why bypass instance lookup

For performance, Python's built-in operations (str, len, +, print) go straight to the type's slot table in C. Skipping instance lookup makes every operation faster — but means dunders MUST be on the class.

Monkey-Patching the Class (vs Instance)

You CAN reassign __str__ on the class — but include self in the lambda since Python passes the instance.

# ❌ Forgot self — TypeError
Stock.__str__ = lambda: "different"
print(str(s))
# TypeError: () takes 0 positional arguments but 1 was given

# ✅ Include self — works
Stock.__str__ = lambda self: "different"
print(str(s))    # 'different'

This is monkey-patching

Modifying a class at runtime — adding, replacing, or removing methods. Python allows it because classes are mutable dicts under the hood.

class Stock:
    """A simple stock with ticker + price."""

    # ──── DUNDER #1: __init__ — constructor ────
    def __init__(self, ticker, price):
        self.ticker = ticker
        self.price = price

    # ──── DUNDER #2: __str__ — human-friendly text ────
    def __str__(self):
        return f"{self.ticker}: ${self.price}"

    # ──── DUNDER #3: __repr__ — debug-friendly text ────
    def __repr__(self):
        return f"Stock(ticker={self.ticker!r}, price={self.price})"

    # ──── DUNDER #4: __eq__ — equality comparison ────
    def __eq__(self, other):
        return self.ticker == other.ticker and self.price == other.price

    # ──── REGULAR INSTANCE METHOD ────
    def is_expensive(self):
        return self.price > 200

    def apply_discount(self, percent):
        self.price = self.price * (1 - percent / 100)
        return self.price
