What is LangChain?

LangChain is a framework for building LLM-powered applications. Its core value is a unified API across providers — swap OpenAI for Claude, or Qdrant for Pinecone, by changing one line. The rest of your pipeline stays identical.

Document → Loader → Splitter → Embeddings → Vector Store → Retriever → LLM → Answer

Each box in that pipeline is a swappable LangChain component. This is what we built in P2 DocTalk.

Package Structure

LangChain split its original monolithic package into sub-packages so you only install what you need. Each provider team (OpenAI, Qdrant, Google) maintains their own package.

from langchain_community.document_loaders import PyPDFLoader

loader = PyPDFLoader("Mavic Drone Manual.pdf")
documents = loader.load()
# documents[0].page_content → text of page 1
# documents[0].metadata    → {'source': 'Mavic...pdf', 'page': 0}

2. Text Splitters Used in P2

Break documents into chunks before embedding. Smaller chunks = more precise retrieval. Overlap ensures answers that straddle a boundary aren't lost.

SplitterWhen to use
RecursiveCharacterTextSplitterGeneral purpose — tries paragraph → line → sentence → word in order
TokenTextSplitterSplit by exact token count (more precise for LLM context limits)
MarkdownHeaderTextSplitterSplit at ## headings — ideal for docs and wikis
HTMLHeaderTextSplitterSplit at <h1>, <h2> tags
CodeTextSplitterSplit at function/class boundaries — preserves code structure
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,    # target characters per chunk
    chunk_overlap=50   # consecutive chunks share 50 chars — boundary context preserved
)
chunks = splitter.split_documents(documents)
# chunks[0].page_content → first 500 chars
# chunks[1].page_content → starts 50 chars before chunk[0] ended

4. Vector Stores Used in P2

Store embeddings and run similarity search. LangChain wraps each one with the same interface — .add_documents() and .similarity_search() work identically across all.

FAISSLocal, in-memoryMeta's library — extremely fast for large corpora
PineconeManaged cloudProduction-grade, serverless, requires API key
PGVectorPostgreSQL extensionGood if you already have Postgres — no new infra
WeaviateOpen-source cloud/localBuilt-in hybrid search (vector + keyword)
from langchain_qdrant import QdrantVectorStore
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams

client = QdrantClient(":memory:")   # no API key, no server
client.create_collection(
    collection_name="docs",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE)
)
vector_store = QdrantVectorStore(client=client, collection_name="docs", embedding=embedding)
vector_store.add_documents(chunks)

results = vector_store.similarity_search_with_score(question, k=3)
# returns list of (Document, float) tuples — doc + cosine score

6. Chains P4

Pre-built pipelines that string together components. A chain takes an input, passes it through multiple steps, and returns a final output.

ChainWhat it does
RetrievalQARAG in one call — retrieves + sends to LLM + returns answer
ConversationalRetrievalChainRAG with chat history — remembers previous turns
LLMChainSimple prompt template → LLM → output
SequentialChainOutput of step N feeds into step N+1
MapReduceChainSummarise long docs — process chunks in parallel, combine results
Important: Chains are the "old" LangChain API (LCEL / legacy). LangGraph (P3) is the modern replacement — more flexible, supports branching, cycles, and human-in-the-loop. Use LangGraph for new projects.

7. Agents & Tools P3 P4

Let an LLM decide which tools to call, in what order, based on the user's goal. The LLM reasons about the task and picks tools — your code doesn't hardcode the sequence.

P1 connection: The tool loop you built manually in agent.py (Claude tool_use) is exactly what initialize_agent does under the hood — but LangChain abstracts the loop. LangGraph (P3) makes it a proper stateful graph.

8. Memory P4

Give chains and agents conversation history across turns. Without memory, every call is stateless — the LLM has no context from previous messages.

ClassWhat it doesTradeoff
ConversationBufferMemoryKeeps full history in RAMSimple but grows without bound
ConversationSummaryMemorySummarises old turns using an LLMSaves tokens, loses some detail
ConversationBufferWindowMemoryKeeps last N turns onlyFixed token cost, forgets old context
VectorStoreRetrieverMemoryStores memories as embeddings, retrieves relevant onesScales well — only recalls what's relevant

LCEL — LangChain Expression Language

LCEL is LangChain's pipe-based syntax for wiring components together. The | operator passes the output of each step as the input to the next — same as Unix pipes, but in Python.

| is standard Python bitwise OR, but LangChain's Runnable base class redefines it via __or__ to mean "chain these steps". Every LangChain component (retriever, prompt, llm, parser) inherits from Runnable.

How invoke flows through the chain

When you call chain.invoke(question), LangChain calls .invoke() on each step automatically — you only call it once. Plain Python functions (like format_docs) are called directly without .invoke().

The fan-out dict pattern

The prompt needs two inputs (context and question) but the chain only receives one (the question string). The dict fan-out splits one input into two named values:

chain.invoke("How do I return the drone to home automatically?")
                    │
          ┌─────────┴──────────┐
          │                    │
          ▼                    ▼
   retriever("How do I...?")   RunnablePassthrough()
          │                    │
          ▼                    │
   [doc1, doc2, doc3]          │
          │                    │
          ▼                    │
   format_docs([doc1,doc2,doc3])│
          │                    │
          ▼                    ▼
   "context string"     "How do I return...?"
          │                    │
          └─────────┬──────────┘
                    ▼
       {"context": "context string",
        "question": "How do I return...?"}
                    │
                    ▼
                 prompt  ← fills {context} and {question} placeholders
                    │
                    ▼
                  llm   ← Claude API call
                    │
                    ▼
           StrOutputParser()  ← extracts plain string from AIMessage
                    │
                    ▼
              answer (string)

RunnablePassthrough() means: take the original input to chain.invoke() and pass it through unchanged into the "question" key. Without it the original question string would be lost after the retriever step.

When to use LCEL: once you understand the manual version (doctalk.py), LCEL makes multi-step chains more readable and composable. LangGraph (P3) builds on this same pipe pattern for stateful agent graphs.

LangChain vs LangGraph

LangChain (Chains)LangGraph
StructureLinear pipeline (A → B → C)Directed graph (nodes + edges, can loop)
StatePassed implicitlyExplicit typed state object
BranchingLimitedFull conditional routing
Loops / retriesNot supportedNative — graph can cycle
Human-in-the-loopAwkwardBuilt-in interrupt/resume
CheckpointingNoYes — resume from any node
When to useSimple RAG, quick prototypesAny multi-step agent workflow
Bottom line: LangGraph is built on top of LangChain core — it's not a replacement for loaders, splitters, embeddings, or vector stores. It replaces the orchestration layer (Chains + initialize_agent). You'll use both together in P3 and beyond.
