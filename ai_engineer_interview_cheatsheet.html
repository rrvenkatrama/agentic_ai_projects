P1 Pydantic v2

What it is

Runtime data validation library. Define a typed schema — Pydantic validates incoming data against it and raises a clear error if it doesn't match.

BaseModel

from pydantic import BaseModel

class WeatherInput(BaseModel):
    city: str               # required
    units: str = "celsius"  # optional, has default
  • Annotations (city: str) stored in __annotations__ at class definition time
  • Pydantic reads those and auto-generates __init__, type checking, and validation
  • Fields end up as instance variables, not class variables

field_validator

from pydantic import field_validator

@field_validator("units")
@classmethod
def check_units(cls, v: str) -> str:
    if v not in ("celsius", "fahrenheit"):
        raise ValueError("must be celsius or fahrenheit")
    return v  # return value is used as the final field value
  • Runs before the object is created — no instance exists yet, so @classmethod is required
  • v = the value being validated
  • Return the value (can transform it, e.g. return v.lower())

ValidationError

from pydantic import ValidationError

try:
    validated = WeatherInput(**raw_input)
except ValidationError as e:
    return f"Invalid arguments: {e}"
  • Raised when type check or field_validator fails
  • Always catch at system boundaries (external data, LLM output)

model_dump()

validated = WeatherInput(city="London", units="celsius")
validated.model_dump()
# → {"city": "London", "units": "celsius"}
  • Converts Pydantic object → plain dict
  • Use **validated.model_dump() to unpack as function kwargs

P1 Claude API — tool_use

How tool_use works (the loop)

You → Claude (messages + tools list)
Claude → stop_reason="tool_use" wants to call a tool
You → execute tool, collect result
You → Claude (messages + tool result)
Claude → stop_reason="end_turn" gives final answer

Repeat if Claude calls multiple tools in sequence.

stop_reason values

ValueMeaning
"end_turn"Claude is done, final answer ready
"tool_use"Claude wants to call one or more tools
"max_tokens"Hit token limit before finishing

Tool schema (JSON Schema)

{
    "name": "get_weather",
    "description": "Get current weather for a city.",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "City name"},
            "units": {"type": "string", "enum": ["celsius", "fahrenheit"]},
        },
        "required": ["city"],
    },
}
description is critical — Claude uses it to decide when to call the tool. required lists mandatory fields; omitted fields use defaults.

Tool result message format

{"role": "user", "content": [
    {
        "type": "tool_result",
        "tool_use_id": block.id,   # must match the tool_use block id
        "content": "result string",
    }
]}
  • Tool results go back as role "user" — not "assistant"
  • tool_use_id links result to the specific tool call Claude made

TOOL_MAP pattern

TOOL_MAP = {
    "get_weather": (get_weather_fn, WeatherInput),
    "calculator":  (calculator_fn,  CalculatorInput),
}

def execute_tool(name, raw_input):
    fn, schema_cls = TOOL_MAP[name]
    validated = schema_cls(**raw_input)         # Pydantic validates
    return fn(**validated.model_dump())          # call with clean args

Single dispatch point for all tools. Keeps validation and dispatch logic in one place.

P1 Streaming (Anthropic SDK)

Non-streaming vs streaming

# Non-streaming — waits for full response
response = client.messages.create(model=..., ...)

# Streaming — prints word by word
with client.messages.stream(model=..., ...) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
    response = stream.get_final_message()  # must be inside the with block
  • get_final_message() must be called inside the with block
  • Returns same object as messages.create() — .stop_reason, .content all available
  • end="" prevents newline per token; flush=True forces immediate output
When to stream: Stream the final text response to the user (better UX). The tool loop itself works the same — stop_reason and content come from get_final_message().

P1 Python Concepts

async/await

async def run_agent(message: str) -> None:
    response = await some_async_call()

asyncio.run(main())   # entry point for async programs
  • async def marks a coroutine — must be awaited or run via asyncio.run()
  • Lets I/O-bound work (API calls) run without blocking

@classmethod vs instance method

class Foo:
    def instance_method(self):    # receives object instance
        ...
    @classmethod
    def class_method(cls):        # receives the class itself
        ...

Used in Pydantic validators because no instance exists yet at validation time.

**kwargs unpacking

def greet(name, greeting="Hello"):
    print(f"{greeting}, {name}")

d = {"name": "Rajesh", "greeting": "Hi"}
greet(**d)   # same as greet(name="Rajesh", greeting="Hi")

Silent function overwrite

def foo(): return "first"
def foo(): return "second"  # silently replaces the first — no error

foo()  # → "second"
Gotcha: Python silently overwrites functions defined with the same name. No error, no warning. The first definition is gone.

P1 System Prompts

What it is

A permanent instruction given to the LLM before the conversation starts. The user never sees it. Sets persona, constraints, tool guidance, output format, and context for the entire session.

Mental model: Like a job briefing you give an employee before their first day. User messages are the work requests that come in after.

Anthropic — separate system parameter

client.messages.create(
    model="claude-opus-4-6",
    system="You are ToolBot, a helpful assistant...",  # ← separate param
    messages=[{"role": "user", "content": "..."}]
)

OpenAI — inside messages list as role "system"

client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "You are ToolBot..."},  # ← first message
        {"role": "user", "content": "..."}
    ]
)

Gemini — passed in GenerateContentConfig

client.models.generate_content(
    model="gemini-2.5-flash",
    contents=contents,
    config=types.GenerateContentConfig(
        system_instruction="You are ToolBot...",  # ← config param
        tools=TOOLS
    )
)

What to put in a system prompt

PurposeExample
Persona"You are a senior financial analyst"
Constraints"Never discuss competitor products"
Tool guidance"Always use the calculator tool for math, never compute yourself"
Output format"Respond in bullet points, max 3 bullets"
Context"You are helping Apple employees with internal IT issues"
Placement gotcha: Anthropic uses a top-level system= parameter. OpenAI uses {"role": "system"} as the first message. Gemini uses system_instruction= inside GenerateContentConfig.

P1 Context Management + Multi-Turn Memory

The problem

If messages = [] is created inside run_agent(), history resets on every call. The agent has no memory of previous turns.

The fix — lift messages to main()

async def run_agent(user_message: str, messages: list) -> None:
    messages.append({"role": "user", "content": user_message})
    # ... rest of loop unchanged

async def main() -> None:
    messages = []   # lives here — persists across all turns
    while True:
        user_input = input("You: ")
        await run_agent(user_input, messages)

Context window limits — simple truncation

MAX_MESSAGES = 20

if len(messages) > MAX_MESSAGES:
    messages = messages[-MAX_MESSAGES:]  # keep most recent N messages
Tool loop gotcha: Tool calls add multiple messages per turn. Truncating mid-tool-call corrupts the history. Safe approach: truncate only after a full turn completes (after end_turn).

Better strategies (for later projects)

StrategyHow
Simple truncationKeep last N messages (P1 approach)
Token-budget trimmingLangGraph trim_messages() — respects token limits
SummarisationUse Claude Haiku to summarise old turns into one message
Sliding windowKeep system + first N + last M messages

P1 Message History Pattern

messages = [{"role": "user", "content": user_message}]

# After Claude responds:
messages.append({"role": "assistant", "content": response.content})

# After tool execution:
messages.append({"role": "user", "content": tool_results})
  • Claude sees the full history on every API call — stateless API, stateful client
  • Never mutate existing messages — always append

Anthropic vs OpenAI — Side by Side

Client setup

# Anthropic
import anthropic
client = anthropic.Anthropic()

# OpenAI
from openai import OpenAI
client = OpenAI()

Tool schema format

# Anthropic
{
    "name": "get_weather",
    "description": "...",
    "input_schema": {           # ← key is "input_schema"
        "type": "object",
        "properties": { ... },
        "required": ["city"],
    }
}

# OpenAI
{
    "type": "function",         # ← extra wrapper
    "function": {
        "name": "get_weather",
        "description": "...",
        "parameters": {         # ← key is "parameters"
            "type": "object",
            "properties": { ... },
            "required": ["city"],
        }
    }
}

Stop / finish reason

# Anthropic
response.stop_reason == "end_turn"    # done
response.stop_reason == "tool_use"    # wants tools

# OpenAI
response.choices[0].finish_reason == "stop"        # done
response.choices[0].finish_reason == "tool_calls"  # wants tools

Accessing tool calls

# Anthropic — iterate response.content blocks
for block in response.content:
    if block.type == "tool_use":
        name = block.name
        args = block.input          # already a dict
        id   = block.id

# OpenAI — iterate message.tool_calls
for tool_call in response.choices[0].message.tool_calls:
    name = tool_call.function.name
    args = json.loads(tool_call.function.arguments)  # JSON string → dict
    id   = tool_call.id

Feeding tool results back

# Anthropic — single "user" message with list of results
messages.append({
    "role": "user",
    "content": [
        {"type": "tool_result", "tool_use_id": id, "content": result}
    ]
})

# OpenAI — one "tool" message per result
messages.append({"role": "assistant", "content": ..., "tool_calls": [...]})
messages.append({"role": "tool", "tool_call_id": id, "content": result})

Streaming

# Anthropic
with client.messages.stream(model=..., ...) as stream:
    for text in stream.text_stream:          # text_stream gives strings directly
        print(text, end="", flush=True)
    response = stream.get_final_message()    # ← get_final_message()

# OpenAI
with client.chat.completions.stream(model=..., ...) as stream:
    for event in stream:                     # yields typed events, not strings
        if event.type == "content.delta":
            print(event.delta, end="", flush=True)
    response = stream.get_final_completion() # ← get_final_completion()
OpenAI streaming events: yields typed event objects, not raw text. event.type == "content.delta" → text chunk via event.delta. Unlike Anthropic, there is no text_stream shortcut.

P1 Gemini (google-genai SDK)

Setup

from google import genai
from google.genai import types

client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

Tool definition — pass Python functions directly

TOOLS = [get_weather, calculator, get_time]
# SDK uses inspect to read: function name, type hints (schema), docstring (description)
Always add docstrings: Without a docstring Gemini only knows the function name. It will guess — and guess wrong on complex tools.
def get_time(timezone: str = "UTC") -> str:
    """Get the current date and time for a given timezone.

    Args:
        timezone: IANA timezone name e.g. 'America/New_York'. Defaults to UTC.
    """

Detect function calls

fn_calls = [p for p in response.candidates[0].content.parts
            if p.function_call and p.function_call.name]

Feed results back

# Append assistant response to history
contents.append(response.candidates[0].content)

# Collect all tool results into one Content block
tool_results = []
for part in fn_calls:
    result = execute_tool(part.function_call.name, dict(part.function_call.args))
    tool_results.append(types.Part(
        function_response=types.FunctionResponse(
            name=part.function_call.name,
            response={"result": result}
        )
    ))
contents.append(types.Content(parts=tool_results, role="user"))

P2 Embeddings

What an embedding is

A list of floats that represents a string's position in semantic space. Vectors with similar meaning point in nearly the same direction.

  • Input: string
  • Output: list[float] — 1536 numbers for text-embedding-3-small

OpenAI embedding API

response = client.embeddings.create(input=text, model="text-embedding-3-small")
vector = response.data[0].embedding   # list[float], length 1536

Local embeddings (no API, no cost)

from sentence_transformers import SentenceTransformer
model = SentenceTransformer("all-MiniLM-L6-v2")  # 384 dimensions
vector = model.encode("some text")   # runs on your machine

Embedding model comparison

ModelDimsTypeCost
text-embedding-3-small1536APIPer token
text-embedding-3-large3072APIPer token
text-embedding-004 (Google)768APIPer token
all-MiniLM-L6-v2384LocalFree
Anthropic Claude—N/ANo embedding API
Hard rule — never mix embedding models. You must use the same model for indexing AND querying. Different models produce vectors in incompatible spaces — mixing makes cosine similarity scores meaningless.

Cosine similarity

def cosine_similarity(a: list[float], b: list[float]) -> float:
    dot = sum(a[i] * b[i] for i in range(len(a)))
    mag_a = math.sqrt(sum(x**2 for x in a))
    mag_b = math.sqrt(sum(x**2 for x in b))
    return dot / (mag_a * mag_b)
  • Measures the angle between two vectors, not distance
  • Dividing by magnitude normalises for document length — measure meaning, not size
  • Score guide: 1.0 identical, 0.8+ very similar, 0.5–0.7 related, <0.3 unrelated

P2 RAG (Retrieval Augmented Generation)

What it solves

LLMs don't know your private documents and can't fit a 100-page PDF in a prompt. RAG retrieves the relevant parts first, then sends only those to the LLM.

INDEXING (once): document → chunk → embed each chunk → store in vector DB
QUERYING (per Q): question → embed → find top-K similar chunks → LLM → answer

Why chunks, not full documents

  • Token limits on embedding APIs
  • A small focused chunk scores higher similarity than a large unfocused one
  • Precision: find 2–3 sentences that answer the question, not the whole document

Chunk size tradeoff

Smaller chunksLarger chunks
More precise retrievalMore context per chunk
May lose surrounding contextMay dilute similarity score
More API calls to embedFewer chunks to manage
Default starting point: 256–512 tokens, 50-token overlap between chunks. Overlap ensures answers that straddle a boundary aren't lost.

LangChain + Qdrant RAG stack (DocTalk)

ComponentPackageRole
PyPDFLoaderlangchain-communityLoads PDF → list of Document objects (one per page)
RecursiveCharacterTextSplitterlangchain-text-splittersSplits pages into chunks
OpenAIEmbeddingslangchain-openaiCalls OpenAI API → 1536-float vector
QdrantClientqdrant-clientCreates/manages in-memory vector store
QdrantVectorStorelangchain-qdrantLangChain wrapper — add/search documents

Full RAG pipeline (doctalk.py pattern)

# 1. Index
loader = PyPDFLoader("doc.pdf")
documents = loader.load()
chunks = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50).split_documents(documents)
client = QdrantClient(":memory:")
client.create_collection("docs", vectors_config=VectorParams(size=1536, distance=Distance.COSINE))
embedding = OpenAIEmbeddings(model="text-embedding-3-small")
vector_store = QdrantVectorStore(client=client, collection_name="docs", embedding=embedding)
vector_store.add_documents(chunks)

# 2. Retrieve
results = vector_store.similarity_search_with_score(question, k=3)

# 3. Format context with citations
context = ""
for doc, score in results:
    context += f"[Page {doc.metadata.get('page','?')}, score: {score:.4f}]\n{doc.page_content}\n\n---\n\n"

# 4. Generate with Claude
response = anthropic.Anthropic().messages.create(
    model="claude-sonnet-4-6",
    system="Answer using only the context provided. Cite page numbers.",
    messages=[{"role": "user", "content": f"Context:\n{context}\n\nQuestion: {question}"}]
)

P3 LangGraph — Core

What it is

A framework for building stateful multi-step agent workflows as directed graphs. Replaces LangChain's old Chains/agents API. Nodes are functions, edges define flow, state is shared across all nodes.

State (TypedDict) → flows through → Nodes (functions) → connected by → Edges

Key concepts

ConceptWhat it is
StateGraph(Schema)Creates the graph. Argument is the state schema — NOT inheritance
TypedDictDefines the shape of the shared state dict
NodePlain Python function: takes full state, returns dict of changed fields only
Fixed edgeadd_edge("a", "b") — always go from a to b
Conditional edgeadd_conditional_edges("a", router_fn) — branch based on state
set_entry_point()Which node runs first
ENDTerminal constant — graph stops when an edge points here
compile()Validates graph, returns a Runnable with .invoke()

LangGraph vs LangChain Chains

ChainsLangGraph
StructureLinear onlyGraph — branch + loop
StateImplicitExplicit TypedDict
Human-in-the-loopAwkwardBuilt-in
CheckpointingNoYes
"Runnable" is not a Python built-in. It's a LangChain/LangGraph concept — any object with .invoke(). graph.compile() returns a Runnable. In plain Python the equivalent is any callable with __call__.

P3 Annotated State — Reducers

By default, when a node returns a field it overwrites the existing value. Annotated lets you specify a merge strategy instead.

from typing import TypedDict, Annotated
import operator

class ResearchState(TypedDict):
    topic: str                                         # overwrite
    search_queries: Annotated[list[str], operator.add] # append (concatenate)
    findings: Annotated[list[str], operator.add]       # append (concatenate)
    summary: str                                       # overwrite

What Annotated actually is

  • Annotated[list[str], operator.add] is a type descriptor object — not a list, not data
  • It only exists at class definition time. At runtime state['search_queries'] is a plain list
  • Python does nothing with the metadata — LangGraph reads it via get_type_hints() at StateGraph() time
  • Uses [] not () — it's a generic type (type parameterisation), not a function call

How LangGraph uses it — reducer map

# Pseudocode — what StateGraph() does internally
for field, type_hint in get_type_hints(ResearchState, include_extras=True).items():
    if is_annotated(type_hint):
        _, reducer = get_args(type_hint)        # extracts operator.add
        reducers[field] = reducer
    else:
        reducers[field] = lambda old, new: new   # default: replace

# After each node returns a delta:
for key, new_value in node_output.items():
    state[key] = reducers[key](state[key], new_value)

operator.add — why it says <built-in function add>

operator.add is implemented in C — Python labels it a "built-in function". It's the + operator as a callable object (you can't pass syntax as an argument).

operator.add(["a"], ["b"])  # same as ["a"] + ["b"]  → ["a", "b"]

Custom reducers

# Merge and deduplicate
seen_urls: Annotated[list[str], lambda a, b: list(dict.fromkeys(a + b))]

# Set union (auto-dedup)
tags: Annotated[set, operator.or_]

# Dict merge (later values win)
metadata: Annotated[dict, operator.or_]

P3 State Immutability Rule

Python passes dicts by object reference, so state inside a node is the live dict. But never mutate it in place — always return a delta.

# CORRECT — return a delta dict
def generate_queries(state: ResearchState) -> dict:
    return {"search_queries": ["q1", "q2"]}

# WRONG — mutating in place
def generate_queries(state: ResearchState) -> dict:
    state["search_queries"] = ["q1", "q2"]  # breaks reducers, checkpointing, parallel nodes
    return {}

Why mutation breaks things:

  • Reducers — operator.add never runs; merge step is bypassed
  • Checkpointing — LangGraph saves state snapshots between nodes; mutation corrupts them
  • Parallel branches — two nodes running concurrently race on the same dict
Mental model: treat state as read-only inside a node. Same pattern as React's setState — never mutate, return what changed.

P3 Conditional Edges — Routing

Router function

def route_summary(state: ResearchState) -> str:
    if len(state['findings']) >= 11:
        return "detailed_summary"
    else:
        return "brief_summary"

Without dict — router return must match node name exactly

graph.add_conditional_edges("research", route_summary)

With dict — translation layer

graph.add_conditional_edges("research", route_summary, {
    "enough": "detailed_summary",    # router returns "enough" → go to "detailed_summary"
    "not_enough": "brief_summary"    # dict is a lookup table — LangGraph reads it
})

Full wiring — two terminal nodes

graph.add_node("detailed_summary", detailed_summary)
graph.add_node("brief_summary", brief_summary)
graph.add_conditional_edges("research", route_summary)
graph.add_edge("detailed_summary", END)
graph.add_edge("brief_summary", END)  # both terminal nodes need their own edge to END
Router is pure logic. It only reads state and returns a string — no API calls, no side effects, no mutations.

P3 Hub-and-Spoke Multi-Agent Architecture

Coordinator fans out to N specialist agents (parallel), aggregator fans in (barrier). Most common enterprise multi-agent pattern.

coordinator ──▶ news_agent ──┐
               ──▶ technical_agent ──┤──▶ aggregator ──▶ END
               ──▶ examples_agent ──┘
  • Fan-out: multiple add_edge() calls from coordinator — LangGraph fires all in parallel
  • Fan-in: multiple edges into aggregator — LangGraph buffers, fires only when ALL branches return
  • Use Annotated[list[Finding], operator.add] — all branches' findings merged automatically
  • Use Pydantic Finding(BaseModel) not list[str] — preserves which agent produced what
  • Coordinator is a pass-through — architecture lives in edges, not coordinator logic

P3 invoke() vs stream()

ModeWhen you see resultsUse case
app.invoke(state)Only after ALL nodes finishScripts, batch jobs
app.stream(state)After EACH node finishesUI progress, monitoring, debug
app.stream(state, mode="events")Node start/end + LLM tokensReal-time UIs
# stream() — yields {node_name: state_delta} after each node
for chunk in app.stream({"topic": "quantum computing", "findings": [], "summary": ""}):
    node_name = list(chunk.keys())[0]
    print(f"[{node_name} done]", chunk)
invoke() vs stream() do NOT change execution. Same graph, same parallelism. Only difference is when results surface to calling code.
stream() gotcha: nodes that return {} (no state update) yield None as the chunk value — not an empty dict. Always guard: if node_data is None: continue
Parallel DuckDuckGo searches: 3 simultaneous requests can trigger rate limiting — one hangs indefinitely. Fix: DDGS(timeout=10)

P3 Agent-as-Consumer (Kafka) — Enterprise Pattern

In-process LangGraph = all agents in one process. Enterprise = agents as independent services on a message bus.

In-process (LangGraph)Event-driven (Kafka/SQS)
All agents die if process crashesIndependent services — fault isolated
Can't scale one agent independentlyScale news_agent × 10, others × 1
Work lost on crashKafka persists messages — replay on restart
New agent = redeploymentNew consumer subscribes to topic, zero downtime
  • Kafka — high throughput, durable (Apple/Netflix/Uber scale)
  • AWS SQS/SNS — AWS-native, easier to operate
  • Redis Streams — lower volume, lower ops overhead
Interview line: "In production I'd decouple these agents via SQS or Kafka for fault isolation, independent scaling, and durability. LangGraph is right for prototyping and moderate-scale deployments."


P3 Checkpointing

LangGraph saves full state after every node. Enables crash recovery, replay, and is required for HITL.

CheckpointerStorageUse case
MemorySaverRAMDev/testing — lost on restart
SqliteSaverSQLite filePoC — persists across restarts
PostgresSaverPostgresProduction — concurrent access
from langgraph.checkpoint.memory import MemorySaver
memory = MemorySaver()
app = graph.compile(checkpointer=memory)
  • Checkpointing ≠ HITL — use alone for crash recovery on long-running graphs
  • HITL always needs a checkpointer — but checkpointing doesn't need HITL
  • SqliteSaver.from_conn_string() returns context manager in newer LangGraph — use MemorySaver for dev

P3 Human-in-the-Loop (HITL)

# 1. Compile with interrupt
app = graph.compile(checkpointer=memory, interrupt_before=["summarise"])
config = {"configurable": {"thread_id": "1"}}

# 2. Phase 1 — run until interrupt point
app.invoke(initial_state, config)

# 3. Inspect paused state
state = app.get_state(config)
findings = state.values["findings"]

# 4. Inject human edits (operator.add merges — list not replace)
app.update_state(config, {"findings": ["my own finding"]})

# 5. Resume — None = load from checkpoint, don't restart
