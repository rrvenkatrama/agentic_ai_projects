What is LangGraph?

LangGraph is a framework for building stateful, multi-step agent workflows as directed graphs. Each step is a node. Data flows between nodes via a shared state object. You define which node runs after which using edges.

It sits on top of LangChain core but replaces the old Chains/agents API with a proper graph-based state machine — supporting branching, loops, checkpointing, and human-in-the-loop.

Install: pip install langgraph — separate from LangChain, maintained by the LangChain team.

LangGraph vs LangChain Chains

State — TypedDict

The state is a plain Python dict with declared types. It is the single source of truth shared across all nodes. Every node reads from it and returns a partial update — only the fields it changed.

from typing import TypedDict

class ResearchState(TypedDict):
    topic: str               # input — the research topic
    search_queries: list[str] # populated by Node 1
    findings: list[str]       # populated by Node 2
    summary: str              # populated by Node 3

TypedDict is just a dict with declared keys and types. It's still a plain dict at runtime — the type hints are for IDE help and LangGraph's schema validation.

Key rule: StateGraph(ResearchState) is NOT inheritance. It passes ResearchState as a configuration argument — LangGraph stores it as a schema to understand the shape of data flowing through the graph.

Nodes

A node is a plain Python function. Signature: takes the full state, returns a dict of only the fields it changed. LangGraph merges the returned dict into the current state automatically.

def generate_queries(state: ResearchState) -> dict:
    queries = [
        f"What is {state['topic']}?",
        f"{state['topic']} latest developments",
        f"{state['topic']} key concepts"
    ]
    return {"search_queries": queries}
    # Only returns what changed — LangGraph merges it into full state
    # topic, findings, summary are unchanged and don't need to be returned
Why return only changed fields? Nodes are single-responsibility. Node 1 only knows about queries — it shouldn't touch findings or summary. LangGraph merges partial updates, so you only return what you changed.

State passing — immutability rule

Python passes dicts by object reference, so state inside a node is the live dict in memory. But never mutate it in place — always return a delta dict.

# CORRECT — return a delta dict
def generate_queries(state: ResearchState) -> dict:
    return {"search_queries": ["q1", "q2"]}

# WRONG — mutating in place
def generate_queries(state: ResearchState) -> dict:
    state["search_queries"] = ["q1", "q2"]  # breaks reducers, checkpointing, parallel nodes
    return {}
Why in-place mutation breaks things:
1. Reducers — operator.add never runs; the merge step is bypassed entirely
2. Checkpointing — LangGraph saves state snapshots between nodes; mutation corrupts them
3. Parallel branches — two nodes running concurrently would race on the same dict

Mental model: treat state as read-only inside a node. Same pattern as React's setState — you never mutate, you return what changed.

Edges

Edges define execution order. Two types:

TypeHow to defineWhen to use
Fixed edgegraph.add_edge("a", "b")Always go from a to b
Conditional edgegraph.add_conditional_edges("a", router_fn)Go to different nodes based on state
Entry pointgraph.set_entry_point("node_name")Which node runs first
Terminal edgegraph.add_edge("last_node", END)Graph stops after this node
graph.set_entry_point("generate_queries")     # first node
graph.add_edge("generate_queries", "research")  # fixed: always go to research next
graph.add_edge("research", "summarise")         # fixed: always go to summarise next
graph.add_edge("summarise", END)               # terminal: stop after summarise

Compile & Invoke

# Build the graph
graph = StateGraph(ResearchState)
graph.add_node("generate_queries", generate_queries)
graph.add_node("research", research)
graph.add_node("summarise", summarise)
graph.set_entry_point("generate_queries")
graph.add_edge("generate_queries", "research")
graph.add_edge("research", "summarise")
graph.add_edge("summarise", END)

# Compile → returns a Runnable object
app = graph.compile()

# Invoke with initial state — all fields must be present
result = app.invoke({
    "topic": "LangGraph state machines",
    "search_queries": [],
    "findings": [],
    "summary": ""
})
# result is the final state dict after all nodes ran

graph.compile() validates the graph (checks all edges point to real nodes, entry point exists, etc.) and returns a Runnable — an object with .invoke(). LangGraph's Runnable is the same base class used by LangChain LCEL components.

"Runnable" is not a Python built-in. It's a LangChain/LangGraph concept — an object that has .invoke(). In plain Python the equivalent is any callable object (one with __call__). LangGraph just standardises on .invoke() as the interface.

Annotated State — Merge Strategy

By default, when a node returns a field, it overwrites the existing value. But for lists you often want to append instead. Annotated lets you specify the merge strategy.

from typing import TypedDict, Annotated
import operator

class ResearchState(TypedDict):
    topic: str                                      # overwrite on update
    search_queries: Annotated[list[str], operator.add]  # append on update
    findings: Annotated[list[str], operator.add]        # append on update
    summary: str                                    # overwrite on update

Annotated[list[str], operator.add] means: "this field is a list of strings, and when a node returns an update, use operator.add (list concatenation) to merge it instead of overwriting."

Without AnnotatedWith Annotated[list, operator.add]
Node returns ["query1"] → replaces existing listNode returns ["query1"] → appended to existing list
Only last node's output survivesMultiple nodes can add to the same list
Fine for scalar fields (topic, summary)Needed when multiple nodes contribute to a list

Annotated — What It Actually Is

Is Annotated[list[str], operator.add] a list?

No. It's a type descriptor object — not a list, not a container of data. It only exists at the class definition level as metadata. The actual values flowing through the graph at runtime are plain Python lists.

import typing, operator
t = typing.Annotated[list[str], operator.add]
print(type(t))            # typing.Annotated[list[str], <built-in function add>]
print(isinstance(t, list)) # False

# What actually lives in the state dict at runtime — plain types:
state = {
    "topic": "LangGraph state machines",  # str
    "search_queries": ["q1", "q2"],         # plain list[str]
    "findings": [],                          # plain list[str]
    "summary": ""                            # str
}
Blueprint analogy: Annotated[list[str], operator.add] is like an annotation on an architectural drawing that says "this pipe uses compression fittings." The annotation is on the drawing — the actual pipe in the building is just a pipe.

Python's part — Annotated

Annotated does nothing at runtime. It's purely a typing construct that stores two things: the real type and the metadata. Python itself doesn't know what operator.add means here — it's just a passenger.

import typing, operator
t = typing.Annotated[list[str], operator.add]
print(typing.get_args(t))
# (list[str], <built-in function add>)
#              ↑ operator.add stored as metadata — Python does nothing with it

LangGraph's part — building the reducer map

When you call StateGraph(ResearchState), LangGraph inspects every field using typing.get_type_hints() and builds a reducer map — one merge function per field:

# Pseudocode of what LangGraph does internally
for field, type_hint in get_type_hints(ResearchState, include_extras=True).items():
    if is_annotated(type_hint):
        _, reducer = get_args(type_hint)    # extracts operator.add
        reducers[field] = reducer             # stores it
    else:
        reducers[field] = lambda old, new: new  # default: replace
FieldReducer stored
topiclambda old, new: new — replace
search_queriesoperator.add — concatenate
findingsoperator.add — concatenate
summarylambda old, new: new — replace

Then after every node returns a delta dict, LangGraph applies the reducer for each field:

# Pseudocode of state merge after each node
for key, new_value in node_output.items():
    state[key] = reducers[key](state[key], new_value)

# For search_queries:
state["search_queries"] = operator.add(["q1","q2","q3"], ["q4","q5"])
# → ["q1","q2","q3","q4","q5"]

# For summary (plain str — replace reducer):
state["summary"] = new_value  # just overwrites

Why <built-in function add>?

operator.add is implemented in C as part of Python's standard library, so Python labels it a "built-in function" rather than a user-defined function. It's just the + operator as a callable object — it exists so you can pass + as an argument (you can't pass syntax directly).

import operator

print(operator.add)   # <built-in function add>  — implemented in C

def my_add(a, b): return a + b
print(my_add)         # <function my_add at 0x...>  — user-defined Python

operator.add(["a"], ["b"])  # same as ["a"] + ["b"]  → ["a", "b"]
Custom reducers: The second argument to Annotated can be any callable. LangGraph just calls it at merge time. You could write your own:

Annotated[list[str], lambda a, b: list(set(a + b))] — merge and deduplicate
Annotated[list[str], lambda a, b: sorted(set(a + b))] — merge, deduplicate, sort

Fan-out / Fan-in — Parallel with a Gate

You can't run two nodes in parallel and force one to wait for the other — that's sequential by definition. But you can fan-out (run two nodes in parallel) and then fan-in (gate the next node until both finish).

start │ ├──────────────────┐ ▼ ▼ generate_queries generate_followup (3 queries) (2 queries) │ │ └────────┬─────────┘ ▼ merge_node ← LangGraph holds here until BOTH complete │ ▼ research 
graph.set_entry_point("start")
graph.add_edge("start", "generate_queries")
graph.add_edge("start", "generate_followup_queries")
graph.add_edge("generate_queries", "merge")
graph.add_edge("generate_followup_queries", "merge")
graph.add_edge("merge", "research")
# LangGraph sees two edges pointing to "merge" — automatically waits for both
Ordering with operator.add in parallel branches: When two nodes run in parallel and both write to an Annotated list field, the final order depends on which result LangGraph processes first. If order matters, run sequentially. If order doesn't matter (e.g. a bag of search queries), parallel + operator.add is fine.
This is the hub-and-spoke pattern used in P3 Ticket #65 — a coordinator fans out to 3 specialist sub-agents, then an aggregator node waits for all 3 before synthesising results.

Conditional Edges & Routing

Fixed edges always go to the same node. Conditional edges let the graph branch — a router function inspects the state and returns which node to go to next.

Router function

def route_summary(state: ResearchState) -> str:
    if len(state['findings']) >= 11:
        return "detailed_summary"
    else:
        return "brief_summary"

The router takes the full state and returns a string — the name of the next node to route to. LangGraph calls it after the source node completes.

Wiring — without dict (simple)

When the router return values exactly match node names, no dict is needed:

graph.add_conditional_edges("research", route_summary)
# router returns "detailed_summary" or "brief_summary" — must match node names exactly

Wiring — with dict (translation layer)

When you want the router to return readable strings that don't match node names:

graph.add_conditional_edges("research", route_summary, {
    "enough": "detailed_summary",   # router returns "enough" → go to "detailed_summary" node
    "not_enough": "brief_summary"    # router returns "not_enough" → go to "brief_summary" node
})
# The dict is a lookup table. LangGraph reads it — not Python, not the dict itself.

Full routing pattern — two terminal nodes

research completes │ ▼ route_summary(state) ← router inspects len(findings) │ ├ - - ≥ 11 - - → detailed_summary → END │ └ - - < 11 - - → brief_summary → END 
graph.add_node("detailed_summary", detailed_summary)
graph.add_node("brief_summary", brief_summary)
graph.add_conditional_edges("research", route_summary)
graph.add_edge("detailed_summary", END)
graph.add_edge("brief_summary", END)
Router is pure logic — no side effects. It only reads state and returns a string. It should never call APIs, mutate state, or do real work. Keep routers as simple conditionals.

Execution Flow — What Happens Inside invoke()

app.invoke(initial_state) called │ ▼ generate_queries reads: state['topic'] = "LangGraph state machines" returns: {"search_queries": ["What is...?", "...latest", "...concepts"]} LangGraph merges → state now has 3 queries │ ▼ add_edge("generate_queries", "research") │ ▼ research reads: state['search_queries'] = [3 items] returns: {"findings": ["Finding for: ...", ...]} LangGraph merges → state now has 3 findings │ ▼ add_edge("research", "summarise") │ ▼ summarise reads: state['findings'] = [3 items] returns: {"summary": "Summary of LangGraph state machines: ..."} LangGraph merges → state now has summary │ ▼ add_edge("summarise", END) │ ▼ invoke() returns final state dict

The state dict is the baton in a relay race. Each node picks it up, reads what it needs, adds its contribution, and passes it on. No node knows about the others — they just read and write the shared state.

graph_intro.py — Full Code

The complete first LangGraph file from P3 ResearchBot. A 3-node linear graph with simulated (fake) node work — no external API calls. Purpose: understand the graph mechanics before adding real web search.

from typing import TypedDict
from langgraph.graph import StateGraph, END

# 1. State schema — shared dict flowing through all nodes
class ResearchState(TypedDict):
    topic: str
    search_queries: list[str]
    findings: list[str]
    summary: str

# 2. Node functions — take full state, return only changed fields
def generate_queries(state: ResearchState) -> dict:
    queries = [
        f"What is {state['topic']}?",
        f"{state['topic']} latest developments",
        f"{state['topic']} key concepts"
    ]
    return {"search_queries": queries}

def research(state: ResearchState) -> dict:
    findings = [f"Finding for: {q}" for q in state['search_queries']]
    return {"findings": findings}

def summarise(state: ResearchState) -> dict:
    summary = f"Summary of {state['topic']}: " + " | ".join(state['findings'][:2])
    return {"summary": summary}

# 3. Build and compile the graph
def build_graph():
    graph = StateGraph(ResearchState)

    graph.add_node("generate_queries", generate_queries)
    graph.add_node("research", research)
    graph.add_node("summarise", summarise)

    graph.set_entry_point("generate_queries")
    graph.add_edge("generate_queries", "research")
    graph.add_edge("research", "summarise")
    graph.add_edge("summarise", END)

    return graph.compile()

# 4. Run it
def main():
    app = build_graph()
    result = app.invoke({
        "topic": "LangGraph state machines",
        "search_queries": [],
        "findings": [],
        "summary": ""
    })
    print(result['summary'])

if __name__ == "__main__":
    main()

Hub-and-Spoke Multi-Agent Architecture

A coordinator node fans out to N specialist sub-agents, each runs independently in parallel, then an aggregator fans in and merges results. One of the most common enterprise multi-agent patterns.

coordinator ──fan-out──▶ news_agent ──┐
                     ──────────▶ technical_agent ──┤──▶ aggregator ──▶ END
                     ──────────▶ examples_agent ──┘

Typed inter-node data with Pydantic

Rather than passing raw strings, sub-agents return structured Pydantic objects. This preserves attribution (which agent produced what) and makes downstream aggregation deterministic.

from pydantic import BaseModel

class Finding(BaseModel):
    agent: str      # "news" | "technical" | "examples"
    title: str
    body: str

class HubState(TypedDict):
    topic: str
    findings: Annotated[list[Finding], operator.add]  # merged across all agents
    summary: str

Fan-out — multiple edges from one node

graph.add_edge("coordinator", "news_agent")
graph.add_edge("coordinator", "technical_agent")
graph.add_edge("coordinator", "examples_agent")

LangGraph fires all 3 in parallel. Each returns {"findings": [Finding(...), ...]}. The operator.add reducer merges all three lists automatically.

Fan-in — implicit barrier synchronization

graph.add_edge("news_agent", "aggregator")
graph.add_edge("technical_agent", "aggregator")
graph.add_edge("examples_agent", "aggregator")

LangGraph sees aggregator has 3 incoming branches. It buffers results and fires aggregator only when all 3 have returned. This is a barrier / Promise.all() equivalent — no explicit wait needed.

Coordinator is a pass-through. Its real role is to be the fan-out point in the graph topology. The architecture lives in the edges, not the coordinator logic.

DuckDuckGo parallel search — rate limiting gotcha

When 3 agents fire simultaneous DuckDuckGo searches, one may hang indefinitely waiting for a response. Always set a timeout:

with DDGS(timeout=10) as ddgs:   # timeout in seconds — prevents indefinite hang
    results = ddgs.text(query, max_results=2)

stream() — None chunk for pass-through nodes

When a node returns {} (empty dict — no state update), LangGraph streams its chunk value as None. Guard against it:

for chunk in app.stream(initial_state):
    node_name = list(chunk.keys())[0]
    node_data = chunk[node_name]
    if node_data is None:   # coordinator returns {} → None in stream
        continue
    # process node_data safely

invoke() vs stream()

Two ways to run a compiled LangGraph app — they differ in when results become visible to your code.

ModeReturnsWhen you see resultsUse case
invoke()Final state dictOnly after ALL nodes completeSimple scripts, batch jobs
stream()Generator of state chunksAfter EACH node completesUI progress, monitoring, debugging
stream(mode="events")Generator of fine eventsNode start/end + LLM token streamReal-time UIs, token-level streaming

invoke() — blocking, get everything at end

result = app.invoke({"topic": "quantum computing", "findings": [], "summary": ""})
print(result['summary'])  # printed only after all nodes finish

stream() — yields after each node

for chunk in app.stream({"topic": "quantum computing", "findings": [], "summary": ""}):
    # chunk = {node_name: {state_delta}}
    node_name = list(chunk.keys())[0]
    print(f"[{node_name} completed]")
    print(chunk)

With hub-and-spoke, stream() output looks like:

{'news_agent':       {'findings': [Finding(agent='news', ...)]}}
{'technical_agent':  {'findings': [Finding(agent='technical', ...)]}}
{'examples_agent':   {'findings': [Finding(agent='examples', ...)]}}
{'aggregator':       {'summary': 'Final merged summary...'}}

Each agent's results appear as soon as that agent finishes. The aggregator still waits for all 3 (barrier), but your app code sees partial results stream in.

invoke() vs stream() do NOT change execution. Both run the same graph with the same parallelism. The only difference is when results surface to your calling code.

Agent-as-Consumer (Kafka) — Enterprise Pattern

In-process LangGraph runs all agents in one Python process. For enterprise-scale systems, agents become independent services communicating via a message bus (Kafka, SQS, RabbitMQ).

producer_agent ──publishes──▶ [tasks topic] ──subscribes──▶ news_agent
                                                        ──subscribes──▶ technical_agent
                                                        ──subscribes──▶ examples_agent

consumer agents ──publish──▶ [findings topic] ──▶ aggregator_agent

Why enterprises use this vs in-process

In-process (LangGraph)Event-driven (Kafka/SQS)
All agents die if process crashesAgents are independent services
Can't scale one agent independentlyScale news_agent × 10, others × 1
Work lost on crashKafka persists messages — agent replays on restart
Adding new agent requires redeploymentNew consumer subscribes to topic, zero downtime

Common enterprise stacks

BusWhen
Apache KafkaHigh-throughput, durable — Apple, Netflix, Uber scale
AWS SQS/SNSAWS-native shops — easier to operate
Redis StreamsLower volume, lower ops overhead
Azure Service BusAzure-native shops

Checkpointing

LangGraph can save the full graph state after every node. These snapshots allow crash recovery, replay from any point, and are required for HITL.

generate_queries → [snapshot]→ research → [snapshot]→ summarise → [snapshot]

Checkpointer options

CheckpointerStorageUse case
MemorySaverPython dict (RAM)Dev/testing — lost on restart
SqliteSaverSQLite filePoC — persists across restarts
PostgresSaverPostgres DBProduction — concurrent access
from langgraph.checkpoint.memory import MemorySaver

memory = MemorySaver()
app = graph.compile(checkpointer=memory)   # snapshots after every node
SqliteSaver API changed in newer LangGraph — from_conn_string() now returns a context manager, not a direct saver. Use MemorySaver for dev. For SQLite persistence use sqlite3.connect() directly and pass to SqliteSaver(conn).

Checkpointing without HITL

Checkpointing is independent of HITL. Use it alone for crash recovery on long-running graphs — research jobs, multi-step pipelines with expensive API calls. HITL always requires a checkpointer, but checkpointing does not require HITL.

Human-in-the-Loop (HITL)

Pauses the graph at a specific node and waits for human input before continuing. Built on top of checkpointing — the paused state is saved to the checkpointer so it can be resumed.

generate_queries → research → [PAUSE] → summarise
                                         ↑
                    human reviews findings, optionally edits 
# 1. Compile with interrupt point
app = graph.compile(checkpointer=memory, interrupt_before=["summarise"])

# 2. Phase 1 — run until interrupt
config = {"configurable": {"thread_id": "1"}}
app.invoke(initial_state, config)   # stops before summarise

# 3. Inspect state at pause point
state = app.get_state(config)
findings = state.values["findings"]

# 4. Optionally inject human edits into checkpoint
app.update_state(config, {"findings": ["my own finding"]})

# 5. Phase 2 — resume (None = load from checkpoint, don't restart)
result = app.invoke(None, config)
invoke(state, config) vs invoke(None, config): Passing a state dict starts fresh. Passing None tells LangGraph to load the latest snapshot for this thread_id from the checkpointer and resume from there. Never pass the state dict again on resume — it would restart from the beginning.

thread_id & config

The config dict is passed to every invoke() call when a checkpointer is attached. It identifies which checkpoint row to read/write.

config = {"configurable": {"thread_id": "1"}}
KeyMeaning
"configurable"LangGraph's namespace for user-defined runtime settings. Other top-level keys: "callbacks", "recursion_limit", "tags"
"thread_id"Identifies this run's checkpoint row. Think: session ID, conversation ID, user ID

The checkpointer SQLite/Postgres table is a shared lookup table. Multiple program instances write to it, each identified by thread_id. In a web app, each user gets their own thread_id — their state is isolated even though they share the same DB.

Reusing thread_id on a second run means LangGraph finds a completed checkpoint and may skip Phase 1. Delete the DB or change thread_id between test runs.

MemorySaver vs SqliteSaver — which to use and where it's defined

# MemorySaver — in-RAM only, defined inside build_graph()
from langgraph.checkpoint.memory import MemorySaver

def build_graph():
    memory = MemorySaver()          # created fresh every run
    ...
    return graph.compile(checkpointer=memory, interrupt_before=["summarise"])

# SqliteSaver — persists to disk across restarts
from langgraph.checkpoint.sqlite import SqliteSaver
memory = SqliteSaver.from_conn_string("checkpoints.db")  # creates .db file
MemorySaverSqliteSaver / PostgresSaver

Survives process restartNo — gone when script exitsYes — persists

research_bot_hitl.py uses MemorySaver — defined on line 55 inside build_graph(). Every time the script runs, a new empty MemorySaver() is created. The thread_id reuse warning only applies when using SqliteSaver/PostgresSaver where checkpoints survive between runs.

Time travel — replay from any snapshot

history = list(app.get_state_history(config))
for snapshot in history:
    print(snapshot.next)           # which node runs next
    print(snapshot.values)         # full state at this point

# Resume from an earlier snapshot
earlier = history[-2]
app.invoke(None, earlier.config)   # re-runs from that point

HITL Deep Dive — update_state, resume, and Annotated reducers

The full HITL sequence

# 1. Compile with checkpointer + interrupt point
checkpointer = MemorySaver()
app = graph.compile(checkpointer=checkpointer, interrupt_before=["human_review"])
config = {"configurable": {"thread_id": "1"}}

# 2. First invoke — runs until interrupt, saves checkpoint
app.invoke({"topic": "quantum computing"}, config)

# 3. Human inspects state, injects a new finding
app.update_state(config, {"findings": [extra]})

# 4. Resume — None = no new input, load from checkpoint
app.invoke(None, config)

How update_state interacts with Annotated reducers

class ResearchState(TypedDict):
    findings: Annotated[list[str], operator.add]   # reducer = concatenation
    summary: str                                    # no reducer = replace

# Annotated field → APPENDS
app.update_state(config, {"findings": ["new finding"]})
# findings = ["old A", "old B", "new finding"]

# Plain field → REPLACES
app.update_state(config, {"summary": "new summary"})
# summary = "new summary"  ← old value gone
Key insight: The Annotated reducer determines update_state behavior — not the value type. A plain list with no reducer is replaced; Annotated[list, operator.add] causes append.

Why invoke(None, config) knows where to resume

  • thread_id is the checkpoint key — same thread_id = same paused run
  • LangGraph saves position (which node stopped) + full state after every node
  • None as input = "don't start fresh, load from checkpoint"
  • Different thread_id = no checkpoint found = starts fresh
