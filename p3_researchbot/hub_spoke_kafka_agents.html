Hub-and-Spoke + Kafka

Agent-as-Consumer design pattern — P3 ResearchBot reference  ·  2026-04-16

What We Built

The same hub-and-spoke research system as hub_spoke.py — coordinator fans out to 3 specialist agents (news, technical, examples), each searches DuckDuckGo, an aggregator collects results — but implemented using Kafka as the communication layer instead of LangGraph.

Each agent is an independent Python script that reads from a Kafka topic and writes to another. They share nothing except the topics. No shared memory, no shared process, no framework.

No LangGraph

This exercise deliberately does not use LangGraph. Kafka replaces what LangGraph was doing:

  • LangGraph edges → producer publishes messages
  • LangGraph barrier → aggregator's seen set
  • HubState TypedDict → JSON messages on topics
  • StateGraph compile → no framework, plain Python

This is closer to how real enterprise systems are built — agents are services that read from queues and write to queues. No framework magic gluing them together.

Kafka Topic

A named, persistent, ordered channel. Messages written to a topic stay there — they don't disappear after being read (unlike a traditional queue). Multiple consumers can read independently from the same topic.

# Topics are created automatically when first written to
# Or explicitly with KafkaAdminClient
producer.send('research-tasks', message)   # creates topic if not exists

Consumer Group ID

group_id is a string label that serves two jobs:

  • Resume tracking — Kafka stores the last read position (offset) per group. On restart, consumer picks up where it left off, not from the beginning
  • Message distribution — controls whether consumers share messages or each get their own copy
c = KafkaConsumer(
    TASKS_TOPIC,
    bootstrap_servers=BROKER,
    group_id=f'{agent_type}-group',   # "news-group", "technical-group", "examples-group"
    ...
)
Changing group_id is like starting fresh — Kafka has no stored offset for the new name, so auto_offset_reset decides where to start.

Fan-out vs Load Balance

The most important Kafka pattern distinction:

Patterngroup_idEach consumer getsUse case
Fan-out (our case)Different per consumerALL messages independentlyEach agent needs every task
Load balanceSame for all consumersSubset of messages (split evenly)Process each message exactly once, spread across workers

Fan-out (different groups)

research-tasks: [msg_news, msg_tech, msg_examples]

news_agent (news-group)   → gets ALL 3 messages, filters for agent="news"
tech_agent (tech-group)   → gets ALL 3 messages, filters for agent="technical"
examples_agent (examples-group) → gets ALL 3 messages, filters for agent="examples"

Load balance (same group)

research-tasks: [msg1, msg2, msg3, msg4, msg5, msg6]

worker_A (workers-group) → gets msg1, msg2
worker_B (workers-group) → gets msg3, msg4
worker_C (workers-group) → gets msg5, msg6

Load balance example: resize 1000 images. Spin up 10 workers in same group — Kafka gives each ~100. No duplicates, no coordination code needed.

Offset & auto_offset_reset

Every message in a topic has a position number called an offset. Kafka tracks which offset each consumer group has read up to.

Topic: [msg0, msg1, msg2, msg3, msg4]
                              ↑
                  news-group last read here (offset=3)
                  → on restart, starts from msg3

auto_offset_reset — what to do when a group has NO stored offset yet (first run, or new group_id):

ValueBehaviourUse when
'earliest'Start from the beginning of the topicWant to process all existing messages on first run
'latest'Start from new messages only (ignore existing)Only care about messages that arrive after starting
Gotcha: Using 'earliest' means re-running agents will pick up old messages from previous runs. This is why the aggregator saw duplicate findings from the previous run — they were still in the topic.

Serializer / Deserializer

Kafka only speaks bytes. Serializers convert Python objects → bytes on send. Deserializers convert bytes → Python objects on receive.

# Producer — dict → JSON string → bytes
KafkaProducer(
    value_serializer=lambda v: json.dumps(v).encode()
)

# Consumer — bytes → JSON string → dict
KafkaConsumer(
    value_deserializer=lambda m: json.loads(m.decode())
)

The lambda parameter (v, m) is just the message value. Could be named anything. Equivalent named function:

def serialize(v):
    return json.dumps(v).encode()

KafkaProducer(value_serializer=serialize)

Producer Pattern

def producer(topic='quantum computing'):
    p = KafkaProducer(
        bootstrap_servers=BROKER,
        value_serializer=lambda v: json.dumps(v).encode()
    )
    for agent_type in ['news', 'technical', 'examples']:
        msg = {'topic': topic, 'agent': agent_type}
        p.send(TASKS_TOPIC, msg)
    p.flush()   # wait until all messages delivered — don't exit early
  • Producer publishes 3 messages and exits — doesn't know or care who's listening
  • flush() — blocks until Kafka confirms receipt. Without it the script may exit before messages are sent

Consumer (Agent) Pattern

def consumer(agent_type):
    c = KafkaConsumer(TASKS_TOPIC, group_id=f'{agent_type}-group',
                       auto_offset_reset='earliest', ...)
    p = KafkaProducer(...)   # also a producer — publishes findings

    for msg in c:            # infinite loop — runs until Ctrl+C
        task = msg.value
        if task['agent'] != agent_type:
            continue        # content-based filtering — ignore other agents' tasks
        # search DuckDuckGo...
        for f in findings:
            p.send(FINDINGS_TOPIC, f.model_dump())
        p.flush()
  • Each agent is both a consumer (reads tasks) and a producer (writes findings)
  • Content-based filtering — sees all messages, acts only on its own type
  • model_dump() — Pydantic v2 method, converts Finding object to plain dict for JSON serialization
  • Runs forever — in production this would be a long-running service

Aggregator Pattern

def aggregator(expected=3):
    c = KafkaConsumer(FINDINGS_TOPIC, group_id='aggregator-group', ...)
    findings = []
    seen = set()
    for msg in c:
        f = Finding(**msg.value)     # ** unpacks dict as keyword args
        if f.agent not in seen:
            seen.add(f.agent)
            findings.append(f)
        if len(seen) >= expected:
            break                    # got one per agent type — done
  • Finding(**msg.value) — msg.value is a dict after deserialization. ** unpacks it as keyword arguments into the Pydantic constructor
  • break — exits the infinite consumer loop once all agent types are represented

seen Set — Deduplication

A simple expected=3 counter is not enough — you might get 2 news findings before any technical or examples arrive, hit the count of 3, and miss the other types entirely.

# Wrong — counts total, not per type
if len(findings) >= 3:
    break    # might have 3 news findings and 0 others

# Right — guarantees one per agent type
seen = set()
if f.agent not in seen:
    seen.add(f.agent)
    findings.append(f)
if len(seen) >= 3:
    break    # guaranteed: one news, one technical, one examples

The seen set also handles duplicate messages from previous runs still in the topic (because auto_offset_reset='earliest' replays old messages for new group IDs).

Kafka on k3s

Single manifest file kafka/kafka.yaml — namespace + deployment + service.

  • Image: apache/kafka:3.7.0 — KRaft mode (no Zookeeper needed)
  • NodePort: 30092 — accessible from Mac at 192.168.1.156:30092
  • Kafka broker port: 9092 internal to the pod
  • KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://192.168.1.156:30092 — tells Kafka to advertise the external address to clients
# Deploy
scp p3_researchbot/kafka/kafka.yaml rajramani@192.168.1.156:~/
ssh rajramani@192.168.1.156 "echo <sudo-pw> | sudo -S kubectl apply -f ~/kafka.yaml"

# Verify
ssh rajramani@192.168.1.156 "echo <sudo-pw> | sudo -S kubectl get pods -n kafka"

# Test connectivity from Mac
python3 -c "from kafka import KafkaAdminClient; a = KafkaAdminClient(bootstrap_servers='192.168.1.156:30092'); print('Connected:', a.list_topics())"

Running the System

Open 5 terminal tabs. Start consumers + aggregator first (they block waiting), then fire the producer.

# Tab 1 — news agent
python p3_researchbot/kafka_agents.py consumer news

# Tab 2 — technical agent
python p3_researchbot/kafka_agents.py consumer technical

# Tab 3 — examples agent
python p3_researchbot/kafka_agents.py consumer examples

# Tab 4 — aggregator (start before producer)
python p3_researchbot/kafka_agents.py aggregator

# Tab 5 — producer (fire last)
python p3_researchbot/kafka_agents.py producer
Order matters: Start the aggregator before the producer. If the aggregator starts after findings are already in the topic, it will still read them (Kafka persists messages) — but starting it first is cleaner and avoids race conditions.

Enterprise Context

In a production enterprise system, each mode would be a separate deployed container:

Script modeProduction equivalent
producerAPI endpoint or scheduled job that triggers research
consumer newsnews-agent-service deployment on k3s, 1-N replicas
consumer technicaltechnical-agent-service deployment on k3s
consumer examplesexamples-agent-service deployment on k3s
aggregatoraggregator-service — writes summary to DB or publishes to results topic
Interview answer: "In a production enterprise setup I'd decouple these agents via Kafka or SQS rather than in-process LangGraph — for fault isolation, independent scaling, and message durability. LangGraph is right for prototyping and moderate-scale deployments where operational simplicity matters more than resilience."
